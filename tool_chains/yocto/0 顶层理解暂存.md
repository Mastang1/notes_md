构建 Yocto 定制化版本，本质上是在主导一个从“零件制造”到“整车组装”的完整自动化工厂。为了建立清晰的顶层心智模型，我们需要严格区分组件级开发（Component Level）**与**系统级集成（System/Image Level）的边界。

以下为你系统性重构 Yocto 的构建逻辑与生命周期。

### 一、 Yocto 顶层心智模型：双线并行的“制造工厂”

在架构设计视角下，Yocto 不是一个线性的单体脚本，而是由**两条截然不同的流水线**和一个**中间仓库**组成的系统：

1. **零件制造线（普通组件 Recipe 流水线）：** 负责拉取源码、交叉编译，并按照严格规范将二进制产物打包成标准化的安装包（RPM/IPK/DEB）。**组件配方的终点，是进入包仓库，而不是进入 Rootfs。**
    
2. **中间件仓库（Package Feeds）：** 存放所有编译打好的标准包。这也是解耦组件开发与镜像集成的核心设计。
    
3. **整机组装线（镜像 Image 流水线）：** 没有任何业务源码。它的唯一职责是拿着一张“物料清单”（BoM，即 `IMAGE_INSTALL`），去仓库里提货，解压拼装成 Rootfs，最后烧录成镜像（Wic/Ext4 等）。

### 二、 核心流程细化对比：组件 vs 镜像的本质区别

基于上述心智模型，我们将详细拆解普通组件（普通 Recipe）与系统镜像（Image Recipe）在构建流程上的核心差异。

#### Phase 1: 普通组件（Recipe）的后半段构建流程

当 `do_compile` 成功后，组件的工作并没有结束，它必须完成自我包装，以符合系统的集成规范。

1. **`do_install` (伪装安装阶段)**
    
    - **动作：** 执行 `make install` 或手写的 `install` 指令。
        
    - **位置：** 文件被复制到组件专属的伪根目录 `${D}`（即工作目录下的 `image/` 文件夹）。
        
    - **架构意义：** 验证软件的目录结构。此时文件仍处于未分类状态，绝对没有进入整个系统的 Rootfs。
        
2. **`do_populate_sysroot` (暴露公共接口)**
    
    - **动作：** 提取 `${D}` 中的 `.h` 头文件、静态库、动态库 `.so`。
        
    - **位置：** 复制到全局的临时编译根文件系统 `sysroot` 中。
        
    - **架构意义：** 仅供其他依赖此组件的模块在**编译期**进行链接（Link），这些文件不会因此进入最终的固件镜像。
        
3. **`do_package` (按规拆包分发)**
    
    - **动作：** Yocto 扫描 `${D}`，剥离二进制调试符号（strip）。随后根据 `FILES:${PN}` 等变量规则，将文件从 `${D}` 分发到不同的临时存放区（如 `packages-split/my-app` 和 `packages-split/my-app-dev`）。
        
4. **`do_package_write_*` (压缩入库)**
    
    - **动作：** 将上一步分离出的各个目录，利用后台工具分别压缩成标准的 Linux 安装包（如 `my-app_1.0_arm64.ipk`）。
        
    - **终点：** 丢入全局仓库 `build/tmp/deploy/ipk/`。**普通组件的生命周期至此彻底结束。**
        

#### Phase 2: 镜像（Image）与 Rootfs 的生成流程

Image 配方（如 `core-image-minimal.bb`）是一类特殊的 Recipe。**它跳过了 Fetch、Compile 等所有源码处理任务**，直接从 `do_rootfs` 开始接管流水线。

1. **`do_rootfs` (组装真实的根文件系统)**
    
    - **触发条件：** Yocto 读取 Image 配方中的 `IMAGE_INSTALL` 列表（例如 `IMAGE_INSTALL += "my-app kernel-module-my-ko"`）。
        
    - **动作：** Yocto 启动一个虚拟的包管理器（如 `dnf` 或 `opkg`），利用配置好的依赖关系，去 `deploy/ipk/` 仓库中提取指定的包及其所有隐式依赖。
        
    - **位置：** 将提取出的包解压到真正的系统组装目录 `${IMAGE_ROOTFS}`（即 Image 工作目录下的 `rootfs/`）。这才是你最终板子上看到的真实文件树。
        
2. **`do_image` (预处理与格式化)**
    
    - **动作：** 对 `${IMAGE_ROOTFS}` 执行最后的修饰（例如处理预加载的系统用户、设置 `/etc/fstab`、生成内核依赖文件 `modules.dep`）。
        
3. **`do_image_<fstype>` (固件封包输出)**
    
    - **动作：** 根据 `IMAGE_FSTYPES` 变量（如 `ext4`, `tar.bz2`, `wic`），调用对应的工具（如 `mkfs.ext4`），将整个 `${IMAGE_ROOTFS}` 目录打包成单体镜像文件。
        
    - **位置：** 存放到最终产物输出目录 `build/tmp/deploy/images/<machine>/`，供开发人员使用。
        

### 总结：包的分发与归属

理解了这两条分离的流水线，就能清晰地解答如何指定打包目标的问题：

- **把文件打进包：** 在**组件 Recipe** 中，使用 `FILES:${PN} += "/path/to/file"`，决定文件归属于哪个 IPK/RPM 零件盒。
    
- **把包打进 Image：** 在**镜像 Image Recipe**（或 `local.conf` 针对该镜像的 append）中，使用 `IMAGE_INSTALL += "my-app"`，决定系统组装时要采购哪些零件。全局的 `local.conf` `append` 会导致所有制造的镜像都强制采购该零件，通常不建议在标准化架构设计中使用。
