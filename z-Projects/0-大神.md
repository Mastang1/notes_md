# 工业级强噪环境离线语音急停控制系统 (V1.0)

## 需求说明 / 详细设计 / 实施策划书

## 第一部分：系统需求规格说明 (SRS)

### 1.1 业务背景与系统定位

系统旨在为强工业噪声（~90dB）环境下的操作工人提供一种“非接触式、解放双手”的辅助安全干预手段。本系统定位于**物理急停按钮的安全冗余层**，通过捕捉特定河北方言口音的关键指令，实现设备的微秒级物理断电。

### 1.2 核心性能指标 (KPIs)

- **端到端响应延迟：** **< 300ms**（从指令发音结束到触发 GPIO 物理继电器的绝对时间差）。
    
- **强噪识别率：** > 95%（在 85dB-95dB 稳态及非稳态机械噪声背景下）。
    
- **误唤醒率 (FAR)：** < 1次/24小时（极其苛刻，宁可拒识不可误识）。
    
- **方言泛化度：** 支持非特定人（多人）的河北方言口音变体及佩戴防尘口罩时的沉闷发音。
    
- **词汇表：** 极小闭集（5个核心词：如开始、停止、加速、减速、复位）。
    

### 1.3 运行环境约束

- **硬件底座：** NXP i.MX6ULL 单核 Cortex-A7 (含配套底板与 GPIO)。
    
- **音频前端：** XMOS XVF3800 DSP 阵列模组（通过 USB 输出 UAC 音频流）。
    
- **操作系统：** Yocto / Buildroot 裁剪版 Linux（无图形界面，无网络依赖）。
    

## 第二部分：系统详细设计 (SDD)

### 2.1 硬件拓扑架构

- **拾音层：** XVF3800 四麦阵列物理硬抗噪声，内部运行 Beamforming（波束成形）指向操作位，开启 BSS（盲源分离）和 AEC（回声消除）。通过 Type-C/USB 线缆输出 16kHz、16bit 的纯净单声道音频流至 i.MX6ULL。
    
- **控制层：** i.MX6ULL 作为主计算单元，仅保留 USB Host 功能与少量 GPIO。
    
- **执行层：** i.MX6ULL 的触发 GPIO 必须通过**高速光耦隔离模块**连接至 24V 工业继电器，最终与机床物理急停开关形成“逻辑或门”电路接入接触器线圈。
    

### 2.2 软件与算法架构

系统采用“旁路物理降噪 + 端到端 Transformer 模型解码”的架构。

- **声学模型：** 采用新一代 `Zipformer-KWS` 架构。相比传统 CNN，Transformer 架构的自注意力机制能更好地捕捉方言中“拖长音”或“倒装”的时序特征。
    
- **训练框架：** 基于 PyTorch 的 `icefall` 框架。
    
- **推理引擎：** `sherpa-onnx`。完全由 C++ 编写，无 Python 运行时开销，直接调用 ONNX Runtime C-API。
    

### 2.3 核心多线程调度模型 (C++ 侧)

主程序摒弃任何会导致系统阻塞的设计，采用无锁环形队列的“生产者-消费者”模型：

1. **生产者线程 (Audio Capture)：** * 利用 ALSA `snd_pcm_readi` 接口，每隔 50ms 从 `hw:1,0` (XVF3800) 读取音频帧。
    
    - 推入预先分配好的静态内存环形缓冲区 (Ring Buffer)。
        
2. **消费者线程 (Inference & Action)：**
    
    - 每次提取 100ms 窗口数据送入 `sherpa-onnx` 流式解码器。
        
    - **防抖逻辑器：** 设计状态机。当“停止”词的 Softmax 置信度超过阈值（如 0.85）且连续命中 2~3 帧时，执行 GPIO 拉低操作。
        
3. **操作系统层保障：** 主进程通过 `sched_setscheduler` 设置为 `SCHED_FIFO`，优先级设为 99，免疫常规系统内核调度的打断。
    

## 第三部分：实施与测试策划 (Execution Plan)

本项目预计执行周期为 **10个工作日**，分四个关键里程碑。

### 里程碑 1：数据工程与基座搭建 (Day 1-2)

- **物理采样：** 前往车间，使用高保真录音笔或 XVF3800 接入 PC，录制 2 小时车间高分贝环境噪声。
    
- **方言采样：** 采集 5-10 位工人的实地河北方言指令（要求包含正常语速、破音吼叫、戴口罩发音），每人每词约 100 遍。
    
- **环境初始化：** 在带有 NVIDIA GPU 的本地 PC 或云服务器上配置 Ubuntu 22.04，搭建 Python 3.8+ 及 PyTorch 环境，克隆 `k2-fsa/icefall` 仓库。
    

### 里程碑 2：模型训练与抗噪附魔 (Day 3-5)

- **数据清洗：** 编写脚本对音频进行切片（约 1-1.5秒/片）和重采样（16kHz）。
    
- **加噪训练：** 在 `icefall` 训练脚本中引入数据增强机制，将车间噪声以随机 SNR（0dB~15dB）混合入干净的方言中进行对抗训练。
    
- **模型固化：** 跑满设定 Epoch 后，验证在测试集上的识别率。达标后，利用导出脚本生成脱离 PyTorch 依赖的 `kws_model.onnx` 和 `tokens.txt` 文件。
    

### 里程碑 3：端侧交叉编译与集成 (Day 6-8)

- **编译环境部署：** 在 PC 上配置 i.MX6ULL 的 `arm-linux-gnueabihf-` 交叉编译工具链。
    
- **框架编译：** 拉取 `sherpa-onnx` 源码，编写 CMakeLists 配置，链接 ALSA 库，一键编译出适用于 ARM 架构的 `.so` 动态库。
    
- **业务代码编写：** 编写 C++ `main.cpp`，实现前文所述的多线程调度、ALSA 音频流读取与 GPIO 寄存器直接映射写入。编译生成可执行文件。
    
- **板载部署：** 将可执行文件、ONNX 模型、动态库拷贝至 i.MX6ULL 开发板。插上 XVF3800 麦克风，确保底层驱动识别正常。
    

### 里程碑 4：实机调优与闭环验收 (Day 9-10)

- **台架测试：** 在办公室内用音响播放车间噪音，测试指令触发的准确性，使用示波器探头抓取声音波形结束到 GPIO 电平跳变的时间差，校验 < 300ms 指标。
    
- **现场防抖调优：** 将系统搬入车间实际工位。根据现场真实的机械轰鸣声，微调 C++ 代码中的“置信度阈值”和“连续判定帧数”，彻底杜绝误触发。
    
- **长测与交付：** 维持系统连续运行 24 小时，监控进程内存占用无泄漏，提交最终验收报告。
