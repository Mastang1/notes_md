## 嵌入式里：架构设计 —— 最常用的 3 条（好记版）

**口诀：「分层隔离 · 质量落点 · 能换能裁」**
_**「关注点分离」（Separation of Concerns）——易变和稳定的东西，因为“变化原因不同”而分到不同层/模块。**_

| #   | 原则（嵌入式里常叫什么）            | 一句话                       | 嵌入式里典型落点                                  |
| --- | ----------------------- | ------------------------- | ----------------------------------------- |
| 1   | **分层 / 关注点分离**          | 变的和不变的分开，别搅在一坨            | 应用 ↔ 驱动核心 ↔ OS 适配 ↔ 板级/HAL ↔ BSP；协议与寄存器分开 |
| 2   | **质量属性优先（NFR 有“架构座位”）** | 实时、安全、功耗、ROM/RAM 要在结构上能兑现 | 中断路径短、WCET 可控；ASIL/MPU 分区；低功耗有明确状态机与唤醒边界  |
| 3   | **可移植 / 可替换 / 可裁剪**     | 换芯片、换 OS、裁功能时少动核心         | 抽象硬件/OS 接口；条件编译/特性开关；核心算法不绑具体 API         |

**记忆卡片（30 秒版）**  
- **分层**：谁管什么，一层一层，别穿透。  
- **质量**：延迟、安全、资源，架构上要有“专人专层”负责。  
- **可换**：平台和 OS 是插件，核心是稳定的。

## 嵌入式里：组件设计 —— 最常用的 3 条（好记版）
_**个人笔记：组件就三个核心东西：接口、依赖、职责；加上限定词就是原则**_

**口诀：「单一职责 · 接口朝外 · 依赖向下」**

| # | 原则（常对应 SOLID 里大家用得最多的几条） | 一句话 | 嵌入式里典型落点 |
|---|------------------------------------------|--------|------------------|
| 1 | **单一职责（SRP）** | 一个模块只为一类变化理由存在 | “队列/缓冲”别写寄存器；“HAL”别写业务状态机 |
| 2 | **显式边界 + 接口隔离（边界/ISP）** | 对外只暴露小接口，内部随便重构 | `.h` 是契约：初始化、读、写、回调；别把头文件打成“大杂烩” |
| 3 | **依赖倒置 / 依赖方向（DIP + 单向依赖）** | 核心依赖抽象，实现细节挂在外围 | 核心调 `os_port`/`hal`，不要反过来；禁止循环依赖 |

**记忆卡片（30 秒版）**  
- **一块一事**：模块越小越硬实时越容易测。  
- **小口对外**：头文件越小，耦合越弱。  
- **核心不依赖底层细节**：底层适配核心，而不是核心 include 板级。
