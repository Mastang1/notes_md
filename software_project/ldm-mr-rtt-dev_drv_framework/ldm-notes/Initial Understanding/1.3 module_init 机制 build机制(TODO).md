# 📝 Linux Kernel: module_init 机制剖析

**Tags:** #Linux/Kernel #Driver_Development #module_init #面试考点

### 📌 Q1: `module_init` 的调用时机是什么？

**A:** 完全取决于模块在 `make menuconfig` 中的编译配置项：

1. ==**静态编译 (`obj-y` / 内置 `*`)**：跟随内核启动调用。发生在内核初始化线程 `kernel_init` 的 `do_basic_setup()` 阶段。==
    
2. ==**动态编译 (`obj-m` / 模块 `M`)**：用户态按需加载时调用。发生在使用终端执行 `insmod` 或 `modprobe` 命令时，内核响应系统调用触发。==
    

### 📌 Q2: `module_init` 的底层实现方式是什么？

**A:** 核心依赖于 C 语言宏定义和链接脚本（Linker Script）魔法，同一套代码自适应两种场景：

1. **静态编译实现**：
    
    - `module_init` 宏展开等同于 `__initcall`。
        
    - 编译器会将修饰的函数指针强制放到特殊的 ELF 数据段（如 `.initcall6.init`）。
        
    - 内核启动时，像遍历数组一样遍历该内存段，逐个执行函数指针，执行完后释放该段内存。
        
2. **动态加载实现**：
    
    - 无法预测内存段，`module_init` 宏会将你的函数重命名为一个标准别名 `init_module`。
        
    - 当加载 `.ko` 文件时，内核的模块加载子系统会解析符号，找到名为 `init_module` 的入口并跳转执行。
        

### 📌 Q3: 调用 `module_init` 的前置条件有哪些？

**A:**

1. **静态编译场景**：
    
    - **核心就绪**：内存管理、中断、进程调度、VFS 等核心基础子系统必须已初始化完毕。
        
    - **层级依赖就绪**：Linux 初始化分级别（0-7级）。`module_init` 默认是第6级（`device_initcall`），它依赖的底层总线驱动（如 PCI/I2C core，通常在更早的 `subsys_initcall`）必须已经先执行完。
        
2. **动态加载场景**：
    
    - **系统就绪**：内核已完全启动，根文件系统已挂载。
        
    - **符号依赖满足**：该模块依赖的其他模块导出的函数（`EXPORT_SYMBOL`）必须已经被提前加载入内核。
        

### 📌 Q4: `module_init` 和“设备-驱动匹配”过程是什么关系？

**A:** 简言之：**`module_init` 负责“报名”，总线负责“相亲”，`probe` 负责“牵手”。**

1. **报名 (`module_init`)**：只负责将驱动的数据结构注册到内核对应的总线上（调用 `xxx_driver_register`），**不涉及任何硬件操作**。
    
2. **匹配 (Bus Match)**：总线核心接到注册请求后，遍历总线上的“未绑定设备链表”，比对 ID table 或设备树 compatible 属性。
    
3. **牵手 (`probe`)**：一旦 `match` 函数比对成功，总线才会去调用驱动代码里的 `probe` 函数。真正的寄存器映射、内存分配、中断申请都在 `probe` 中进行。
