
### 1. 设备号获取的所有相关结构体、子结构体（内核的“商标注册局”）

设备号的管理核心是为了防止全系统的驱动编号发生冲突 。在内核中，这一职责由 `char_device_struct` 结构体和 `chrdevs` 数组承担 。

#### 原始结构体定义

```c
// 定义在 fs/char_dev.c 中
// 全局设备号哈希表 (包含 255 个槽位，用于加快查找)
static struct char_device_struct *chrdevs[CHRDEV_MAJOR_HASH_SIZE];

// 具体的设备号注册节点
struct char_device_struct {
    struct char_device_struct *next; // 链表指针：用于解决哈希冲突，将挂在同一个槽位的节点串起来
    unsigned int major;              // 主设备号 (如 244)
    unsigned int baseminor;          // 起始次设备号
    int minorct;                     // 该驱动申请的连续次设备号数量
    char name[64];                   // 驱动的名称 (如 "ipc-shm")
    struct cdev *cdev;               // 指向对应的 cdev 对象 (注：现代内核主要靠 cdev_map 路由，此指针多为历史遗留或辅助)
};
```

- **功能描述**：它是“设备号占用凭证”。当你调用 `alloc_chrdev_region()` 动态摇号时，内核会创建一个 `char_device_struct` 实例，记录你的主次设备号和名字，并将其挂入 `chrdevs` 数组中 。它只负责“占坑”，不负责后续的通信数据路由 。
    
- ==**关系描述**：多个 `char_device_struct` 通过 `next` 指针形成单向链表，被 `chrdevs` 数组的槽位（按主设备号哈希）所管理 。==
    

### 2. VFS的文件管理、节点管理等所有相关结构体（内核的“户籍与通信录”）

VFS（虚拟文件系统）屏蔽了底层硬件差异。用户态看到的是文件路径，内核中对应的是 `inode` 和 `file` 结构体 。

#### 原始结构体定义

```c
// 定义在 include/linux/fs.h 中

// 1. 物理文件的“身份证/户口本”
struct inode {
    umode_t                 i_mode;     // 文件类型与访问权限 (标识它是一个字符设备文件)
    dev_t                   i_rdev;     // 【核心！】真实的设备号 (包含主、次设备号)
    struct cdev             *i_cdev;    // 运行时缓存指针：open被调用后，指向你注册的 cdev 对象
    
    // ... 省略大量其他通用文件系统属性 (如 uid, gid, size 等) ...
};

// 2. 一次“操作会话”的记录
struct file {
    struct path             f_path;     // 包含 dentry (目录项，记录文件路径名)
    struct inode            *f_inode;   // 指向该文件对应的 inode
    const struct file_operations  *f_op; // 【核心！】操作合同：决定当前 read/write 执行什么代码
    void                    *private_data; // 【核心！】私有数据口袋：驱动开发者用来存放自定义结构体的指针
    
    // ... 省略读写偏移量、标志位等 ...
};
```

- **功能描述**：
    
    - `inode` 静态代表一个物理存在的文件节点（如 `/dev/ipc-shm`），它肚子里存着设备号 `i_rdev` 。
        
    - `file` 动态代表进程打开文件的一次会话。每次 `open` 生成一个，`close` 销毁。它记录了操作函数集 `f_op` 和驱动私有上下文 `private_data` 。
        
- **关系描述**：一个 `inode` 可以对应多个 `file`（同一个设备被多个进程同时打开）。在 VFS 的 `open` 流程中，内核会提取 `inode` 里的 `i_rdev` 去查找驱动，找到后将驱动的 `file_operations` 赋值给 `file->f_op` 。
    

### 3. 设备号和文件路径关联的结构体（内核的“114 黄页与路由枢纽”）

这是内核从 VFS 跨越到具体驱动的最核心枢纽。它负责将 VFS 传来的“设备号（数字）”映射为“真实的 `cdev` 结构体（对象）” 。

#### 原始结构体定义

C

```
// 定义在 drivers/base/map.c 和 fs/char_dev.c 中

// 全局唯一的字符设备路由哈希表
static struct kobj_map *cdev_map;

// kobj_map 结构体：通用的内核对象映射表
struct kobj_map {
    struct probe {                      // 路由探针/字典条目
        struct probe *next;             // 哈希冲突链表指针
        dev_t dev;                      // 【核心 Key】设备的“门牌号” (设备号)
        unsigned long range;            // 设备号覆盖范围
        struct module *owner;           // 所属内核模块 (用于引用计数)
        void *data;                     // 【核心 Value】泛型指针，被强制塞入你的 struct cdev 首地址
    } *probes[255];                     // 基于主设备号的 255 个哈希链表槽位
    
    struct mutex *lock;                 // 互斥锁，保护并发查表与注册
};
```

- **功能描述**：`cdev_map` 是一张极其高效的哈希查找表。`struct probe` 就像黄页上的一条名片记录 。
    
- **关系描述**：当你调用 `cdev_add()` 时，内核创建一个 `probe`，其 `dev` 记录设备号，`data` 强绑定你的 `cdev` 指针 。当 VFS 拿着 `inode` 里的设备号来查表时，命中 `probe` 后，提取 `data` 即可拿到你的驱动对象 。
    

### 4. 字符设备作为面向对象的相关结构体和子结构体（你的“店铺”与“操作手册”）

这是驱动开发者直接操作的数据结构。通过在自定义结构体中内嵌 `struct cdev`，Linux 内核以 C 语言实现了面向对象（OOP）的继承与多态机制 。

#### 原始结构体定义

C

```
// 定义在 include/linux/cdev.h 中

// 内核认识的字符设备“基类”
struct cdev {
    struct kobject kobj;                  // 内核对象基础类 (提供引用计数、sysfs 树形结构支持)
    struct module *owner;                 // 指向 THIS_MODULE，防止设备在使用时模块被卸载
    const struct file_operations *ops;    // 【核心！】指向你实现的 read/write 等函数集
    struct list_head list;                // 链表节点
    dev_t dev;                            // 绑定的起始设备号
    unsigned int count;                   // 绑定的设备号数量
};

// ==========================================
// 驱动开发者自定义的“子类” (以 ipc-shm 为例)
// ==========================================
struct ipc_shm_dev {
    struct cdev cdev;                     // 继承 cdev 基类 (不强制放首位，靠 container_of 定位)
    dev_t dev_num;                        // 驱动申请到的设备号
    void *shm_base_addr;                  // 共享内存映射地址 (私有数据)
    int tx_irq_id;                        // 发送中断号 (私有数据)
    int rx_irq_id;                        // 接收中断号 (私有数据)
};

// 定义在 include/linux/fs.h 中
// 物流合同/虚函数表
struct file_operations {
    struct module *owner;
    int (*open) (struct inode *, struct file *);
    ssize_t (*read) (struct file *, char __user *, size_t, loff_t *);
    ssize_t (*write) (struct file *, const char __user *, size_t, loff_t *);
    int (*mmap) (struct file *, struct vm_area_struct *);
    int (*release) (struct inode *, struct file *);
    // ... 省略 ioctl, poll, flush 等 ...
};
```

- **功能描述**：
    
    - `cdev` 是字符设备在内核中的核心控制块，承载了驱动的具体能力（`ops`）。
        
    - `ipc_shm_dev` 是带有私有上下文的业务实体。
        
    - `file_operations` 是一组函数指针，是 VFS 和底层驱动之间的“接口契约” 。
        
- **关系描述（多态实现）**：
    
    1. 实例化：`cdev_init()` 将你写的 `file_operations` 绑定到 `cdev.ops` 。
        
    2. 上下文获取：当 `open` 触发时，驱动通过 `container_of` 宏，从传入的 `inode->i_cdev` 指针反向计算出 `struct ipc_shm_dev` 的首地址，进而拿到 `tx_irq_id` 等私有数据 。
        
    3. 狸猫换太子：VFS 提取 `cdev.ops` 替换掉 `file->f_op`，后续读写直接调用你实现的方法 。
