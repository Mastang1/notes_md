
这是一份为你量身定制的 Kbuild Makefile 心智模型与背诵笔记。

记住一个核心隐喻：**外置内核模块的 Makefile 具有“双重人格”，而 `KERNELRELEASE` 就是切换人格的触发开关。**

### 🧠 Kbuild 两次遍历机制（心智模型笔记）

#### 1. 黄金骨架（闭眼能默写版本）

Makefile

```
ifneq ($(KERNELRELEASE),)
    # -----------------------------------
    # 人格 A：【内核打工人】 (第二次遍历)
    # -----------------------------------
    obj-m := my_module.o
    
else
    # -----------------------------------
    # 人格 B：【本地引路人】 (第一次遍历)
    # -----------------------------------
    KDIR ?= /lib/modules/$(shell uname -r)/build
    PWD  := $(shell pwd)

default:
    $(MAKE) -C $(KDIR) M=$(PWD) modules
endif
```

#### 2. “双重人格”原理解释

- **唯一判断标准：`KERNELRELEASE` 变量。**
    
    - 你敲击键盘执行 `make` 时，你处于本地普通终端，此时没有这个变量。
        
    - 只有 **Linux 内核的顶层 Makefile** 才拥有并能赋予这个变量。
        
- **第一次遍历（人格 B：本地引路人）**
    
    - **触发场景：** 你在当前目录手动敲击 `make`。
        
    - **判断结果：** `KERNELRELEASE` 为空，走 `else` 分支。
        
    - **唯一任务：“去总部搬救兵”。** 执行 `make -C $(KDIR)` 瞬间跳转到内核源码树总部，并用 `M=$(PWD)` 留下一张字条：“总部配置好编译环境后，请按这个地址回来找我。”
        
- **第二次遍历（人格 A：内核打工人）**
    
    - **触发场景：** 内核总部收到字条，带着所有的编译器参数（如 CPU 架构、交叉编译链）、并且**带着赋值好的 `KERNELRELEASE`**，重新杀回你的代码目录。
        
    - **判断结果：** `KERNELRELEASE` 不为空，走 `ifneq` 分支。
        
    - **唯一任务：“只管干活”。** 此时 Makefile 完全闭嘴，只向内核暴露出 `obj-m := my_module.o`，内核的 Kbuild 机制会接管剩下的所有 `.c` 到 `.ko` 的编译工作。
        

#### 3. Yocto 视角下的极简补充

如果你**只在 Yocto 环境中**编译这个模块（比如写了 `inherit module`），你的心智模型可以进一步简化到极致：

- **Yocto 是个“保姆”**：Yocto 的 `module.bbclass` 在底层**直接替你完成了“人格 B”的工作**。它在调用 make 时，会在命令行强行注入内核路径和各种环境变量。
    
- **你可以怎么偷懒**：如果这个 Makefile 只在 Yocto 里跑，你连 `ifneq` 都可以不写！**直接保留“人格 A”的部分**（即 `obj-m := my_module.o`），它照样能完美运行。写上完整的骨架，仅仅是为了保证这份代码将来脱离 Yocto 也能在普通 Ubuntu 电脑上手动编译。
