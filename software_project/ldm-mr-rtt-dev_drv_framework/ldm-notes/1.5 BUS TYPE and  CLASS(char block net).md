
# 🧠 【架构师笔记】Linux 驱动框架核心流转与架构

## 核心心智模型：Linux 驱动的“两层皮”哲学

在 Linux 复杂的底层中，为了极致的解耦，设备驱动被划分为职能完全分离的两个部门：

- **【后勤部】通用设备模型 (LDM)**：负责解决“物理管理”问题（电源、热插拔、拓扑关系、内存映射）。
- **【业务部】分类设备模型 (Class)**：负责解决“数据流转”问题（怎么读写、按字节还是按数据包）。 当你在写驱动时，你通常是**“两线作战”**：先向后勤部注册拿内存，再向业务部注册给用户提供读写接口。

---

## 模块一：后勤部 (LDM 通用设备模型) 的权力边界

LDM 的核心是“总线-设备-驱动”铁三角，其职责边界泾渭分明，绝不越权。

### 1. Bus (总线)：平台方 / 红娘

- **职能本质**：没有任何硬件操作能力，只负责规则和撮合。
- **核心动作**：
    - **管名册**：维护 `klist_devices` (设备链表) 和 `klist_drivers` (驱动链表)。
    - **配对 (Match)**：执行总线特有的匹配规则（如比对 Device Tree 中的 `compatible` 字符串）。
    - **广播**：设备插拔时向用户空间上报 `Uevent`。

### 2. Driver (驱动)：接管方 / 专职保姆

- **职能本质**：Match 成功后，接管特定物理设备的生命周期。
- **核心动作**：
    - **入职接管 (Probe)**：申请内存映射 (`ioremap`)、申请中断号 (`request_irq`)。
    - **退休注销 (Remove)**：释放物理资源。
    - **电源管理**：处理 `Suspend` (休眠保存状态) 和 `Resume` (唤醒恢复)。

## 模块二：业务部 (Class 分类模型) 的三大底层兵种

Linux 将纷繁复杂的物理世界，根据“数据流转方式”高度抽象为三大经典操作架构。

### 兵种 A：字符设备 (Character Device) —— 【直通水管模型】

- **物理本质**：数据按**字节流**顺序吞吐，无巨型缓存。
- **核心载体**：`struct cdev` + `struct file_operations`。
- **用户接口**：通过 `/dev/ttyS0` 等节点，暴露出标准的 POSIX 接口 (`open/read/write`)。
- **数据流转**：App $\rightarrow$ 虚拟文件系统 (VFS) 指针跳转 $\rightarrow$ 你的 `fops->write` $\rightarrow$ 寄存器。（同步且直接）。

### 兵种 B：块设备 (Block Device) —— 【仓储物流模型】

- **物理本质**：数据必须按**固定块**（如 4KB）读写，支持随机访问，硬件极其慢。
- **核心载体**：`struct gendisk` + `request_queue` (请求队列)。
- **认知重塑**：驱动中**没有**直接的 `read/write` 函数！
- **数据流转**：App写数据 $\rightarrow$ 内存页缓存 (Page Cache) $\rightarrow$ 内核 I/O 调度器 (排序、合并请求) $\rightarrow$ 塞入 `request_queue` $\rightarrow$ 驱动提货交给 DMA。

### 兵种 C：网络设备 (Network Device) —— 【快递包裹模型】

- **物理本质**：面向**数据包 (Packet)** 收发，彻底打破“万物皆文件”规则。
- **核心载体**：`struct net_device` + `struct net_device_ops`。
- **认知重塑**：在 `/dev/` 下**没有设备节点**，完全对接到 TCP/IP 协议栈。
- **数据流转**：App 调用 `socket()` $\rightarrow$ TCP/IP 协议栈打包 $\rightarrow$ 生成内核网络终极结构 `sk_buff` $\rightarrow$ 调用驱动的 `ndo_start_xmit()` 扔出网线。
