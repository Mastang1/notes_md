
## 1. 核心结构(4大基础结构) + 初始注册FS实例、open泛化

你的意图非常清晰：你不仅希望从宏观上理解 VFS（虚拟文件系统）的“接口与实现分离”理念，更要求**深入 Linux 内核源码级**，扒出那些藏在代码仓库深处的核心结构体和 API 函数，并且要求精准标注它们所在的文件路径。

### 一、 费曼图式导入：VFS 的架构定性

在没有 VFS 之前，如果你要读取硬盘（ext4）、网络（NFS）或内存（sysfs），你需要写三套完全不同的应用程序代码。 VFS 的出现，就是为了实现**“契约的绝对统一”**。

- **VFS 静态分层（物流大厦）**：最上层是用户态的 POSIX 标准接口（快递单）；中间层是 VFS 核心枢纽（分拣中心与查号台）；底层是各种具体的文件系统驱动（不同的物流专线，如 ext4 专线、sysfs 专线、cdev 专线）。
- **核心理念**：VFS 自身**不保存**任何实际数据，它只提供一套抽象的数据结构（面向对象中的纯虚基类）和通用的路由算法（路径名解析、缓存查找）。

### 二、 VFS 核心结构体展示（架构的“四大基石”）

为了将字符串路径（如 `/dev/ipc-shm`）转化为底层设备的动作，VFS 依赖四个核心结构体（C 语言实现的面向对象封装）。所有的定义均可在 Linux 内核树的 `include/linux/fs.h` 或相关头文件中找到。

|结构体名称|所在头文件路径|费曼隐喻|核心元素展示与功能描述|
|:--|:--|:--|:--|
|`struct file_system_type`|`include/linux/fs.h`|**物流专线营业执照**|记录文件系统类型（如 "ext4", "sysfs"）及其挂载（mount）方法。内核启动时注册。|
|`struct super_block`|`include/linux/fs.h`|**物流分拨中心（实例）**|代表一个已挂载的具体文件系统实例。保存了磁盘的整体信息（如块大小）以及操作 `inode` 的虚函数表 `s_op`。|
|`struct inode`|`include/linux/fs.h`|**货物物理身份证**|代表文件系统中的一个**物理文件**（或设备节点）。包含文件大小、权限，如果是设备文件，最核心的是包含 `i_rdev`（设备号）。|
|`struct dentry`|`include/linux/dcache.h`|**路径导航路标**|目录项。用于将字符串名字（如 "ipc-shm"）映射到具体的 `inode`，构成目录树并加速查找。|
|`struct file`|`include/linux/fs.h`|**本次会话的购物篮**|代表进程与文件的一次**动态会话**。包含当前读写偏移量 `f_pos`，以及最重要的操作契约 `f_op` 和私有口袋 `private_data`。|

#### 源码级核心元素剖析（以 `inode` 和 `file` 为例）：

```c
/* 路径: include/linux/fs.h */
struct inode {
    umode_t                 i_mode;     /* 辨别是普通文件、目录还是字符/块设备 */
    const struct file_operations *i_fop;/* 默认的文件操作手册 */
    dev_t                   i_rdev;     /* 极其核心：若是设备文件，存放主次设备号 */
    union {
        struct block_device *i_bdev;
        struct cdev         *i_cdev;    /* 若为字符设备，向底层绑定的指针 */
    };
    /* ... 锁、权限、时间戳等略 ... */
};

/* 路径: include/linux/fs.h */
struct file {
    struct path             f_path;     /* 包含 dentry 和 mnt，定位文件所在挂载点 */
    struct inode            *f_inode;   /* 指向背后的物理身份证 */
    const struct file_operations *f_op; /* 多态的核心！VFS调用读写时查阅的函数表 */
    loff_t                  f_pos;      /* 当前会话的读写偏移量 */
    void                    *private_data; /* 留给底层驱动保存多实例上下文的终极口袋 */
    /* ... 引用计数、权限标志等略 ... */
};
```

### 四、 核心场景 2：Open 场景的详细路由流程 (UML 序列图)

**场景描述**：这是 VFS 最波澜壮阔的时刻。用户空间仅传入一个字符串路径（如 `/dev/ipc-shm`），VFS 需要完成**“路径解析 -> Inode提取 -> 会话建立 -> 契约替换（狸猫换太子）”**的全部过程。

**涉及核心 API 与路径**：

1. `do_sys_open()` / `do_sys_openat2()` (路径: `fs/open.c`)
2. `path_openat()` -> `link_path_walk()` (路径: `fs/namei.c`，核心路径解析)
3. `vfs_open()` (路径: `fs/open.c`)
4. `fd_install()` (路径: `fs/file.c`)

#### 知识锚点映射 (对比 `mr-library` 的 `mr_dev_open`)

在 `mr-library` 中，`mr_dev_open` 是在内存里直接用 `strcmp` 匹配字符串找到设备节点。 但在 Linux 中：

- **字符串查找的脏活**被 `fs/namei.c` 中的 `link_path_walk` 揽下了，它不仅查缓存（`dentry`），没有的话还要去读磁盘/内存文件系统提取 `inode`。
- **操作函数的绑定**被 `fs/open.c` 中的 `vfs_open` 和底层的兜底函数联合完成，最终实现了 `file->f_op` 的偷天换日。
