这份关于 Yocto 路径变量的终极笔记，是你以后编写、调试和阅读开源代码配方时的“指南针”。一旦掌握了这些变量，那些看起来像天书一样的 `.bb` 文件就会变得非常清晰。

以下为你整理的 **Yocto 核心路径与工程实践终极笔记**：

### 📓 附录 A：Yocto 核心路径变量速查表

为了防止在绝对路径和相对路径中迷失，Yocto 强制使用以下三类变量来定位空间。

#### 1. 沙盒生命周期目录（你在哪里炒菜）

这些变量全部位于 Yocto 深邃的 `build/tmp/work/...` 目录下，是配方执行过程中的专属隔离区。

|**变量名**|**核心作用与说明**|**你的视角 (类比)**|
|---|---|---|
|**`${WORKDIR}`**|当前配方的绝对工作根目录。所有的源码、补丁、临时脚本都在这下面。|你的专属独立厨房操作台。|
|**`${S}`**|Source Directory。源码解压存放地。`do_compile` 默认在这里找 Makefile。|案板（放洗好切好的菜）。|
|**`${B}`**|Build Directory。编译输出地。如果分离编译（如 CMake），生成的 `.o` 在这里。|炒锅（加热食材的地方）。|
|**`${D}`**|Destination Directory。虚拟根文件系统。`do_install` 必须且只能把文件放这里。|盘子（没放进盘子的菜，上不了桌）。|

#### 2. 系统标准目标路径（你要把菜装在哪里）

在 `do_install` 往 `${D}` 中拷贝文件时，**绝对禁止**硬编码写 `/usr/bin`，必须使用以下变量，以保证在不同架构下的兼容性。

|**Yocto 变量**|**对应的最终系统绝对路径**|**通常用来存放什么**|
|---|---|---|
|**`${bindir}`**|`/usr/bin`|用户级可执行程序（App）。|
|**`${sbindir}`**|`/usr/sbin`|系统级管理工具、守护进程。|
|**`${libdir}`**|`/usr/lib`|动态库（`.so` 文件）。|
|**`${sysconfdir}`**|`/etc`|配置文件（`.conf`、`.json`、`.ini`）。|
|**`${includedir}`**|`/usr/include`|C/C++ 头文件（`.h`）。|
|**`${datadir}`**|`/usr/share`|架构无关的数据（如字体、图标、系统默认只读配置）。|
|**`${localstatedir}`**|`/var`|运行时可变数据（如日志 `/var/log`、PID 文件）。|
|**`${base_libdir}`**|`/lib`|基础底层库、内核模块（`/lib/modules/`）。|
|**`${systemd_unitdir}`**|`/lib/systemd/system`|Systemd 开机自启服务文件（`.service`）。|

#### 3. 工程相对寻址锚点（你在哪里找食材）

当需要引用外部代码或同级目录的文件时，用于动态定位的锚点。

|**锚点变量**|**指向的绝对路径说明**|
|---|---|
|**`${THISDIR}`**|指向**当前正在被解析的 `.bb` 或 `.bbappend` 文件**所在的绝对目录。极常用于 `externalsrc` 定位同级源码。|
|**`${TOPDIR}`**|指向**你执行 `source oe-init-build-env` 和敲击 `bitbake` 命令**所在的那个 `build` 目录。|
|**`${COREBASE}`**|指向 **Yocto (Poky / OpenEmbedded-Core) 源码树的顶级根目录**。|

### 💡 深度解答：`${COREBASE}` 等变量是默认的还是自定义的？

这在工程中非常容易混淆，明确区分它们能避免很多诡异的路径错误：

- **默认/内置变量（Yocto 原生提供，开箱即用）：**
    
    - `${COREBASE}`、`${TOPDIR}`、`${THISDIR}` 以及上面的所有目录变量（`${S}`, `${D}`, `${bindir}` 等）都是 Yocto 的核心组件 `bitbake` 引擎在启动时**自动计算并注入**的。你在任何标准的 Yocto 工程中都可以直接使用它们。
        
- **高度定制化变量（如 `${BSPDIR}`，需谨慎）：**
    
    - 如果你在使用 NXP (Freescale)、TI 或 NVIDIA 提供的 Yocto SDK，你经常会在他们的文档或配方中看到类似 `${BSPDIR}` 的变量。
        
    - **这是自定义的！** 这些芯片原厂通常使用 Google 的 `repo` 工具来管理成百上千个 git 仓库。为了方便，他们会在顶层的 `conf/bblayers.conf` 或环境初始化脚本中，手动强行定义一个 `BSPDIR` 变量，用来指向整个大工程的最顶层。
        
    - **风险提示：** 如果你把带有 `${BSPDIR}` 的配方直接拷贝到一个纯净的官方 Yocto 工程中，编译会直接报错找不到该变量。
        

### ⚠️ 生产环境红线警告：为什么极度不推荐 `externalsrc` 入库？

`externalsrc` 是一个让你在本地“爽快”调试的神兵利器，但如果将其写死在正式的 meta-layer 中提交给团队，它将是一场灾难。

**1. 破坏了 CI/CD 的“可重复构建”原则**

`externalsrc` 本质上是指向了开发者电脑硬盘上的一个特定文件夹（无论你用相对路径还是绝对路径）。当你的代码推送到 GitLab/Gerrit，触发服务器上的 Jenkins/GitLab CI 自动构建镜像时，CI 服务器根本找不到你定义的那个本地目录，构建必定崩溃。

**2. 摧毁了 Yocto 的 SSTATE 缓存机制**

Yocto 之所以能在第二次编译时飞快，是因为它会对源码的哈希值、补丁进行校验，如果没变，直接复用缓存。而 `externalsrc` 绕过了 Yocto 的标准下载和校验流程。Yocto 无法有效追踪外部文件夹里代码的细微变动，经常会导致“你明明改了代码，但编译出来的系统还是老代码”的诡异玄学现象。

**3. 污染了依赖图谱**

团队协作中，任何人拉取了你们公司的 meta-layer，他的 `build` 目录旁必须严格按照你设定的相对路径放置特定的源码文件夹，这带来了极高的沟通和维护成本。

#### ✅ 工业级的最佳实践（The Yocto Way）

- **开发调试期：** 尽情使用 `devtool modify / add`（底层自动使用 `externalsrc`），在本地快速验证功能。
    
- **转正入库期：** 调试完成后，执行 `devtool finish`。或者手动将代码提交到公司内网的 Git 服务器上，然后在 `.bb` 配方中将 `externalsrc` 删除，老老实实地改回：
    
    代码段
    
    ```
    SRC_URI = "git://git.yourcompany.com/project/my-daemon.git;protocol=ssh;branch=main"
    SRCREV = "${AUTOREV}"  # 或者指定特定的 commit hash
    ```
