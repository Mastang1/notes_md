### 1. LDM 基础：设备与驱动（资产与操作员）

在 LDM 的世界观里，Linux 强行将“硬件信息”和“软件代码”拆分开了，实现了极度优雅的解耦。

#### 1.1 基础设备结构体：`struct device` (具体的硬件资产)

它代表了挂在系统上的一块真实的物理硅片，或者一个虚拟的逻辑设备。

C

```c
// 定义在 include/linux/device.h 中
struct device {
    struct kobject kobj;          // 【核心基类】挂载到 /sys 目录下的基石，自带引用计数
    struct device *parent;        // 【树形节点】指向父设备（比如某个USB设备挂在USB控制器下）
    
    struct device_private *p;     // 【隐藏的暗箱】存放内核私有数据的指针（里面包含了各种链表！）
    
    struct bus_type *bus;         // 指向该设备挂载的“总线” (如 I2C, SPI, Platform)
    struct device_driver *driver; // 匹配成功后，指向来接管该设备的“驱动程序”
    
    void *platform_data;          // 硬件私有数据（比如板级配置的 GPIO 引脚号等）
    
    dev_t devt;                   // 【终极桥梁！】存放设备号！这就是 LDM 与 cdev 产生关联的唯一纽带！
    struct class *class;          // 该设备所属的“行业协会”（稍后讲）
    
    void (*release)(struct device *dev); // 析构函数，当引用计数为 0 时调用
};
```

- **费曼隐喻：** 这是一台**挖掘机**。它有自己的出厂编号，有它的上级归属（归哪个工区管），同时它身上有一个标志牌 `devt`（设备号），告诉大家如果在商业街（`/dev`）想找它，请拨打这个号码。
    

#### 1.2 基础驱动结构体：`struct device_driver` (持证上岗的操作员)

它代表了一段纯粹的软件逻辑代码。

C

```c
// 定义在 include/linux/device.h 中
struct device_driver {
    const char *name;             // 驱动的名字，用于和 device 的名字进行字符串匹配！
    struct bus_type *bus;         // 该驱动隶属于哪种总线
    
    struct driver_private *p;     // 【隐藏的暗箱】包含驱动绑定的所有设备链表
    
    // 【核心操作：相亲与分手】
    int (*probe) (struct device *dev);  // 当总线发现匹配的 device 时，调用此函数执行初始化
    int (*remove) (struct device *dev); // 当 device 被拔出或模块卸载时调用
    
    // 电源管理的回调函数
    int (*suspend) (struct device *dev, pm_message_t state);
    int (*resume) (struct device *dev);
};
```

- **费曼隐喻：** 这是持有**挖掘机驾驶证的司机**。他不需要知道挖掘机明天在哪，他只要在总线（人才市场）上等着。一旦总线发现有一台名字匹配的挖掘机（`device`）注册了，就会调用他的 `probe()` 函数，让他去上班。
    

### 2. Class 相关的结构体（行业协会）及链表深度剖析

根据你的要求，我们重点扒一扒 `class` 以及里面复杂的**链表结构**。

**为什么要引入 Class？**

在总线（Bus）层面，设备是按物理连接分类的（比如 USB 总线、I2C 总线）。但对用户来说，我不管你是 USB 鼠标还是蓝牙鼠标，你都是“输入设备（Input）”。`class` 就是为了提供这种**高级别的、按功能划分的逻辑视图**（对应 `/sys/class/` 目录）。

#### 2.1 类的核心结构体：`struct class`

C

```c
// 定义在 include/linux/device/class.h 中
struct class {
    const char *name;                   // 类的名字 (如 "tty", "input", "net")
    
    struct module *owner;
    const struct attribute_group **class_groups; // 在 /sys/class/xxx 目录下创建的属性文件
    const struct attribute_group **dev_groups;   // 为属于该类的所有设备自动创建的属性文件
    
    // 【关键点】在早期的 Linux 内核中，链表直接暴露在这里（struct list_head devices）。
    // 但在现代内核中，为了防止驱动开发者乱动链表，内核将其封装到了 subsys_private 中！
    struct subsys_private *p;           // 指向类/子系统的私有数据包装器
    
    // 类的事件钩子（比如发送 uevent 给用户空间的 udevd，用于动态创建 /dev 节点）
    int (*dev_uevent)(const struct device *dev, struct kobj_uevent_env *env);
    char *(*devnode)(const struct device *dev, umode_t *mode);
};
```

#### 2.2 必须强调的链表节点：深挖 `struct subsys_private`

现代内核采用了极度严谨的 OOP 封装。我们把 `class->p` 剖开，来看看里面到底是怎么用链表把设备串起来的：

C

```c
// 定义在 drivers/base/base.h 中 (注意：这是内核底层私有头文件，不对外暴露！)
struct subsys_private {
    struct kset subsys;                 // 代表 /sys/class/xxx 那个目录本身
    
    // 【核心链表 1】挂载了所有属于该 Class 的设备！
    struct klist klist_devices;         
    
    // 【核心链表 2】挂载了该 Class 下的所有接口
    struct klist klist_interfaces;      
    
    struct class *class;                // 回指上层的 class 结构体
};

// klist 是一种自带自旋锁 (spinlock) 的并发安全双向链表
struct klist {
    spinlock_t k_lock;                  // 保护链表并发访问的锁
    struct list_head k_list;            // 【真正的内核标准双向链表头部】
    void (*get)(struct klist_node *);
    void (*put)(struct klist_node *);
};
```

- **费曼隐喻：** `class` 是一个“行业协会”（比如全国出租车协会）**。协会有一个上了锁的**花名册（`klist_devices` 链表）。只要有一台新出租车（`device`）注册并声明属于这个协会，内核就会在内存里生成一个节点，插进这个花名册链表里。当你敲下命令 `ls /sys/class/tty` 时，内核其实就是在遍历这条链表，把名字打印在屏幕上！

### 顶级工程师的终极笔记：

在实际的驱动开发中（如你研究的 `ipc-shm`），你经常会看到下面这种“三连击”代码：

1. `alloc_chrdev_region(&dev_num...)` -> 从商标局申请了设备号。
    
2. `cdev_add(...)` -> 把操作手册（读写代码）通过设备号挂到了黄页（`cdev_map`）上。
    
3. **`device_create(my_class, NULL, dev_num, NULL, "ipc-shm")`** -> **这就是 LDM 登场的时刻！** 它在 `my_class` 的链表中插入了一个新的 `device` 节点，并且内核守护进程（`udev/mdev`）会捕捉到这个动作，**自动在你的 `/dev/` 目录下创建一个名为 `ipc-shm` 的物理设备文件！** 这样你就再也不用手动执行 `mknod` 了。
