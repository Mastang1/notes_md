`device_add()` 是内核设备驱动模型中的“注册中枢”。任何设备（无论是真实的物理硅片，还是纯软件虚拟出来的逻辑设备）想要被内核真正接纳、被用户空间看到，都必须经过这一道关卡。

### 1. `device_add()` 的功能及分支（物理设备 vs 逻辑设备）

**核心功能：**

`device_add()` 的主要职责是将一个已经初始化的 `struct device` 结构体正式加入到内核的设备层次结构（Kobject/Kset 系统）中。它的具体工作包括：

1. **构建 sysfs 层级**：调用 `kobject_add()` 在 `/sys` 文件系统中创建对应的目录。
    
2. **建立关联软链接**：在总线（bus）、类（class）和父设备目录之间创建互相指向的 symlink。
    
3. **暴露属性文件**：创建设备相关的属性文件（如 `uevent`, `dev` 等）。
    
4. **触发驱动匹配**：调用 `bus_probe_device()`，让总线去寻找是否有匹配的 `device_driver`，如果有则执行驱动的 `probe` 函数。
    
5. **发送 uevent**：调用 `kobject_uevent()` 向用户空间（如 `udevd` 或 `mdev`）发送 `KOBJ_ADD` 事件。
    

**物理设备与逻辑设备的分支区分机制：**

`device_add()` 函数内部并没有类似于 `if (is_physical)` 这样的硬编码分支。相反，内核通过 `struct device` 内部挂载的指针（`bus`, `class`, `parent`）来进行优雅的面向对象式区分：

- **物理设备（基于总线）**：
    
    - **特征**：通常存在硬件探测机制，挂载在真实的物理总线上。其 `dev->bus` 指针不为空（例如指向 `pci_bus_type` 或 `usb_bus_type`），并且通常有一个明确的 `dev->parent`（比如挂在某个桥接器或控制器下）。
        
    - **分支体现**：`device_add()` 中会执行 `if (bus) { bus_add_device(dev); }`，这会在 `/sys/bus/<bus_name>/devices/` 下创建该设备的软链接，并强烈依赖 `bus_probe_device()` 来绑定驱动。
        
- **逻辑设备/虚拟设备（基于类或无硬件实体）**：
    
    - **特征**：通常由纯软件创建（如 loopback 网卡、tty 终端、或是为了暴露纯字符设备节点而创建的封装设备）。它们的 `dev->bus` 可能为空，但 `dev->class` 不为空（例如指向 `tty_class`）。或者，它们被显式地挂载在内核的虚拟总线（Platform Bus 常用于设备树设备，但也常被用作“伪总线”来挂载不具备自动枚举能力的设备）。
        
    - **分支体现**：如果 `dev->class` 存在，`device_add()` 会调用 `device_add_class_symlinks(dev)`，在 `/sys/class/<class_name>/` 下创建对应的层次结构。对于完全没有 `parent` 的顶级虚拟设备（如 `/sys/devices/virtual/`），`device_add()` 会调用 `setup_parent()` 将其父节点重定向到 `virtual_device_parent` 目录下。
        

### 2. 实际代码中的调用场景与 UML 序列图

在内核中，驱动开发者极少直接调用 `device_add()`，通常是通过子系统提供的封装函数（如 `device_register`, `platform_device_register`, `device_create` 等）间接调用。

#### 场景一：平台设备注册（Platform Device，最常见的物理/SOC设备）

系统启动时解析设备树（Device Tree），针对设备树中的节点创建 `platform_device` 并注册。

- **调用链**：`of_platform_populate()` -> `of_platform_device_create_pdata()` -> `platform_device_add()` -> `device_add()`。
    

#### 场景二：基于物理总线的热插拔/枚举设备（如 PCI, USB）

当插入一个 USB 设备时，USB Hub 驱动检测到端口变化，枚举设备并注册。

- **调用链**：`hub_port_connect()` -> `usb_alloc_dev()` -> `usb_new_device()` -> `device_add()`。
    

#### 场景三：逻辑设备/字符设备节点的动态创建

驱动程序（如字符设备驱动）在完成底层初始化后，希望在 `/dev` 下生成设备节点，会调用 `device_create()`。

- **调用链**：驱动自身初始化 -> `device_create()` -> `device_create_groups_vargs()` -> `device_register()` -> `device_initialize()` + `device_add()`。

### 3. 执行 `add` 后在 VFS 的哪些挂载位置添加节点？

当 `device_add()` 成功执行完毕后，内核的虚拟文件系统（VFS）中主要会有两个挂载点发生变化：**`sysfs`** 和 **`devtmpfs`**。

#### 1. Sysfs (`/sys` 挂载点) —— 最主要的作用域

`device_add()` 会在 `/sys` 中反映出设备的拓扑结构和分类：

- **真实物理层次（绝对路径）**：
    
    - 节点创建于：`/sys/devices/` 目录下。
        
    - 例如：`/sys/devices/pci0000:00/0000:00:1f.2/ata1/host0/target0:0:0/0:0:0:0/` (一个SATA硬盘的物理层级)。
        
    - 如果是没有父设备的逻辑设备，通常在 `/sys/devices/virtual/<class_name>/<dev_name>`。
        
- **按总线分类（软链接）**：
    
    - 节点创建于：`/sys/bus/<bus_name>/devices/<dev_name>`。
        
    - 这是一个符号链接，指向 `/sys/devices/` 下的真实路径。
        
- **按功能类分类（软链接）**：
    
    - 节点创建于：`/sys/class/<class_name>/<dev_name>`。
        
    - 例如：`/sys/class/net/eth0` 或 `/sys/class/block/sda`，这也是一个指向 `/sys/devices/` 的软链接。
        
- **按设备号分类（软链接）**：
    
    - 如果设备有关联的设备号（`devt`），会在 `/sys/dev/char/<major>:<minor>` 或 `/sys/dev/block/<major>:<minor>` 创建指向对应设备的软链接。
        

#### 2. Devtmpfs (`/dev` 挂载点)

如果你在内核中开启了 `CONFIG_DEVTMPFS`（现代Linux发行版默认开启）：

- 在 `device_add()` 执行过程中，如果传入的 `struct device` 中包含有效的 **主设备号和次设备号 (`dev->devt`)**。
    
- `device_add()` 会在发出 uevent 之前，调用内核内部的 `devtmpfs_create_node(dev)`。
    
- **结果**：直接在内核态于 **`/dev/`** 目录下创建实际的设备节点文件（例如 `/dev/sda` 或 `/dev/ttyS0`），其类型（块设备 `b` 或 字符设备 `c`）和权限也会被自动设置好。
    
- _(补充说明：如果内核没有开启 devtmpfs，那么 `/dev/` 下的节点则是由用户空间的 `udevd` 守护进程在收到上述第四步的 Netlink `KOBJ_ADD` 事件后，读取 `/sys/` 下的信息并在用户态调用 `mknod` 创建的。)_
