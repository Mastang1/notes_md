

**【费曼式降维：`.dtsi` 的本质是什么？】** 在 C 语言中，我们有 `.h`（头文件）和 `.c`（源文件）。在面向对象编程中，我们有“基类”和“派生子类”。 设备树完美套用了这种思想：

- **`.dtsi` (Include)**：相当于**“基类”或“公共组件”**。它描述的是芯片原厂（SoC）内部固定不变的物理硅片拓扑（比如 CPU 核心数、芯片内部 UART/SPI 控制器的物理基地址）。
- **`.dts` (Source)**：相当于**“派生子类”或“板级实例化”**。它描述的是具体的某一块 PCB 开发板（Board），它通过 `#include` 引入原厂的 `.dtsi` 基类，然后根据板子上的实际连线，追加或覆盖属性。

以下是为你整理的便于查阅和记忆的指导笔记。

# 🧠 【架构师笔记】Device Tree (.dtsi/.dts) 标准化开发指南

## 一、 文件组织与依赖拓扑 (File Organization & Dependencies)

设备树的文件组织是一棵极其严谨的**“继承树”**，遵循“从通用到具体”的原则。

1. **核心骨架 (`skeleton.dtsi`)**：定义了最基础的地址长度映射法则（如 `#address-cells`）。
2. **芯片原厂级 (`<soc_name>.dtsi`)**：
    - **职能**：定义某款 SoC（如 `stm32mp151.dtsi`）内部的所有外设控制器的资源（`reg` 寄存器地址、`interrupts` 中断号）。
    - **关键原则**：因为原厂不知道客户会把哪个外设引脚接出来，所以在 `.dtsi` 中，所有外设节点的默认状态必须是 **`status = "disabled";`**。
3. **开发板产品级 (`<board_name>.dts`)**：
    - **职能**：这是最终被编译成 `.dtb` 二进制文件的源头。
    - **动作**：包含对应的 `.dtsi`。针对这块开发板实际连线的硬件（如 I2C 总线上挂了触摸屏），将该控制器的状态复写为 **`status = "okay";`**，并追加板级特有的配置。

## 二、 必须遵循的“契约” (Bindings)

写设备树绝不是凭感觉捏造属性名，系统内核有严格的“宪法”来校验你的数据。

1. **契约所在位置**：Linux 内核源码树的 `Documentation/devicetree/bindings/` 目录下存放着所有的规范说明。
2. **契约形式**：目前主要采用 YAML 格式（以前是 TXT）。它规定了每个 `compatible` 暗号对应的节点规则。
3. **遵守原则**：
    - **Required（必填项）**：比如 `reg` 和 `interrupts`。如果不按契约提供，编译时报错，或者驱动 `probe()` 时直接提取失败导致宕机。
    - **Optional（可选项）**：比如 `dmas`（DMA通道）或特定硬件的 `clock-frequency`。

## 三、 具体开发协约与规则指导 (Rules & Guidelines)

作为驱动工程师，编写和修改设备树时必须将以下规则刻入肌肉记忆：

1. **绝对隔离动态逻辑**：设备树纯粹是一份静态的“物理资产清单”。绝不能在里面包含“如何初始化”、“向寄存器写什么值”等操作流程。业务逻辑永远留在 C 语言的 Driver 中。
2. **标签（Label）引用法则**：
    - 在定义 `.dtsi` 基类时，必须为每个核心节点打上 Label（如 `uart1: serial@40011000`）。
    - 在定义 `.dts` 子类时，**不要**按着原层级结构重新嵌套写一遍，而是直接使用引用语法 `&uart1 { ... };` 进行跨层级覆盖。这能极大降低代码嵌套深度。
3. **地址空间的“翻译”规则 (`ranges`)**：
    - 当设备的物理总线发生跨越层级（比如从 CPU 内存映射总线跨越到挂载的 PCI 总线）时，必须在父节点定义 `ranges` 属性，明确告知内核：子节点的地址段如何映射到 CPU 的真实物理地址上。

## 四、 费曼实战 Demo：从 SoC 基类到 Board 子类

我们用一段极简的 Demo，来看看驱动工程师的实操流转。

### 1. SoC 级别的基类定义 (芯片原厂维护：`my_soc.dtsi`)

_职责：描述不可变的物理硅片资产，默认闭门谢客。_

```
// my_soc.dtsi
/ {
    #address-cells = <1>;
    #size-cells = <1>;

    // 内部外设总线
    soc {
        compatible = "simple-bus";
        ranges; // 地址透传映射

        // 为串口控制器打上标签 "uart1"
        // 节点命名规范：<设备类型>@<物理基地址>
        uart1: serial@40011000 {
            compatible = "vendor,my-uart";  // 契约：驱动匹配的暗号
            reg = <0x40011000 0x400>;       // 契约：基地址与长度
            interrupts = <37>;              // 契约：硬件中断号
            status = "disabled";            // 规则：基类中默认禁用！
        };
    };
};
```

### 2. Board 级别的实例化 (驱动工程师开发：`my_board.dts`)

_职责：描述特定的绿色 PCB 开发板，包含基类，并接通需要的硬件线路。_

```
// my_board.dts
/dts-v1/;
#include "my_soc.dtsi"  // 继承 SoC 基类

/ {
    model = "My Company Smart Gateway Board";
    compatible = "vendor,my-board", "vendor,my-soc";
};

// 黄金法则：利用 dtsi 中定义的标签 &uart1 直接覆写/追加属性
&uart1 {
    status = "okay";               // 动作1：使能它（打开开关）
    current-speed = <115200>;      // 动作2：追加板级特有的配置参数
    pinctrl-names = "default";
    pinctrl-0 = <&uart1_pins_a>;   // 动作3：关联这块板子上为串口分配的具体 GPIO 引脚
};
```

**【架构师结语】** 通过 `.dtsi` 与 `.dts` 的配合，设备树完美实现了**“硬件配置的面向对象设计”**。芯片原厂只需维护极少数的 `.dtsi`（定义规则与基础资产），而千万个下游厂商基于这些基类，通过标签引用（`&label`）优雅地重写属性，生成了无数个 `.dts`（产品实现）。掌握这份笔记，你已经具备了应对任何复杂 Linux/RTOS 硬件抽象层的实操框架能力。
