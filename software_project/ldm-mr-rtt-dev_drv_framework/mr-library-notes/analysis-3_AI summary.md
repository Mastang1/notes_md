# 嵌入式设备驱动框架（MR）底层逻辑与架构笔记

## 核心基建：为什么要引入驱动框架？

在裸机开发中，业务代码直接与具体的硬件寄存器或 HAL 库（如 `HAL_UART_Transmit`）强绑定。一旦更换芯片，需要重写成百上千处业务逻辑。

驱动框架的本质是一个**“沙漏模型”**的中间商。

- **向上（应用层）：** 提供“统一且永不改变”的标准 API（`open` / `read` / `write` / `ioctl`）。应用层只需知道外设的“字符串名字”（如 `"serial1"`），实现**位置无关性**。
    
- **向下（驱动层）：** 强制收编五花八门的芯片底层驱动，将“硬件的易变性”隔离在底层。
    

---

## 架构拆解：框架的“三层脱壳”模型

整个框架从抽象到具体，严格分为三个物理与逻辑层级：

### 1. Device Management (设备管理层：基础设施)

- **核心机制：侵入式链表 (Intrusive List)**
    
    框架不关心具体设备是什么，它只管理一个包含 `struct mr_dev`（基类）的全局链表。这种“数据包含节点”的设计极大地节省了内存。
    
- **核心机制：自动初始化 (Auto Initialization)**
    
    利用编译工具链的 **Linker Section（链接器段）** 魔法。通过宏定义为初始化函数打上标签（如 `section.1`），链接器在编译时将其按顺序紧凑地排列在内存中。系统启动时只需一个简单的 `for` 循环即可完成底层注册，彻底消灭了臃肿的 `main()` 函数中的硬编码初始化。
    

### 2. Device Model (设备模型层：标准策略与OS服务)

- **核心机制：C 语言的面向对象 (OOP in C)**
    
    由于 C 语言没有 `class`，框架通过**“结构体嵌套”**实现继承。`struct mr_dev`（基类）必须放在 `struct mr_serial`（子类）的**绝对第一顺位**。这样内存地址完全重合，基类指针与子类指针可以安全强转。
    
- **职责划定：纯软件逻辑与策略 (Policy)**
    
    这一层提取了某类硬件的“最大公约数”。例如：串口在物理上是异步的，因此框架在 `serial` 模型中引入了 Ring Buffer（环形缓冲区）作为固有的私有数据成员，用来弥合“极速 CPU”与“慢速外设”的鸿沟。这一层负责缓冲管理、线程阻塞/唤醒等纯操作系统层面的逻辑。
    

### 3. Driver Adaption (驱动适配层：硬件机制实现)

- **核心机制：虚函数表 (V-Table)**
    
    通过包含函数指针的结构体 `struct ops` 实现多态。底层工程师的核心工作，就是编写特定芯片（如 NXP/STM32）的 HAL 调用包，然后将函数地址赋给 `ops->write` 等指针。
    
- **职责划定：纯硬件控制 (Mechanism)**
    
    这一层“没有智商，只有肌肉”。它不关心数据排队或线程调度，只负责死磕芯片手册，将 RAM 中的单字节准确推入诸如 `USART1->DR` 这样的物理地址中。

## 运行时魔法：两级解耦与指针流转 (Runtime Double Dispatch)

当应用程序调用一行 `mr_dev_write("serial1", data)` 时，底层发生了一次极其优雅的“桥接模式”跨越：

1. **查表寻址：** 框架首先充当全局路由器，拿着字符串 `"serial1"` 在侵入式链表中匹配，找到位于 RAM 中的特定设备实例（`dev` 指针）。
    
2. **第一级脱壳（调用模型层）：** 框架执行 `dev->ops->write(...)`。此时进入通用设备模型层（如 `serial.c`），执行诸如塞入 Ring Buffer 等独立于硬件的抽象逻辑。
    
3. **第二级脱壳（调用驱动层）：** 模型层在合适时机，调用底层注册的 `drv->ops->transmit(dev, ...)`，触发真正的硬件 HAL 级发送。

## 内存视角的终极洞察：代码共享与状态隔离

在面向对象的设计思想下，如果板子上有两个相同的硬件（如 UART1 和 UART2），它们的内存关系如下：

- **行为逻辑（代码）在 ROM 中只有一份：** 无论是 UART1 还是 UART2，发送数据的时序逻辑是完全相同的。底层驱动代码（如 `stm32_uart_write`）编译后变成机器指令，在 Flash/ROM 中永远只有一份共享代码。
    
- **物理状态（数据）在 RAM 中是多份实例：** UART1 和 UART2 的寄存器基地址、波特率、Ring Buffer 是绝对隔离的。因此在 RAM 中会存在两个独立的子类结构体实例（`uart1_dev`, `uart2_dev`）。
    
- **上下文隔离的枢纽 —— `dev` 指针：** 底层共享代码之所以知道当前该操作哪个串口，是因为框架在每次调用 `ops` 函数指针时，都会将当前的 `dev` 指针作为**第一个参数**传入（完美模拟了 C++ 的 `this` 指针）。底层代码通过这个 `dev` 指针，就能精准反向读取到该特定外设的私有基地址和状态。
