# 🛠️ 核心基建：底层结构体源码全景图

在调用任何 API 之前，我们必须先看清楚内核里有哪些“表单”和“账本”。请死死盯住这些结构体里的**核心注释**。

### 1. 设备号管理账本 (Namespace)

C

```c
// fs/char_dev.c - 占坑用的“商标注册局”
static struct char_device_struct *chrdevs[255]; // 哈希数组

struct char_device_struct {
    struct char_device_struct *next; // 哈希冲突链表
    unsigned int major;              // 主设备号 (如 244)
    unsigned int baseminor;          // 起始次设备号
    int minorct;                     // 连续次设备号数量
    char name[64];                   // 驱动名 (cat /proc/devices 可见)
};
```

### 2. CDEV 路由中枢 (Routing)

C

```c
// drivers/base/map.c - 内核的“114黄页查号系统”
static struct kobj_map *cdev_map;

struct kobj_map {
    struct probe {
        struct probe *next;
        dev_t dev;                   // 【查找键】：设备号
        void *data;                  // 【核心映射】：强制指向你的 cdev 首地址！
    } *probes[255];
};

// include/linux/cdev.h - 你的“实体店铺”基类
struct cdev {
    struct kobject kobj;                  // 内核对象（提供引用计数）
    const struct file_operations *ops;    // 【核心】：绑定的业务函数 (open/read)
    dev_t dev;                            // 绑定的设备号
};
```

### 3. LDM 挂牌系统 (Sysfs & Uevent)

C

```c
// include/linux/device/class.h - “行业协会”门面
struct class {
    const char *name;                   // 协会名 (如 "ipc_shm_class")
    struct subsys_private *p;           // 【暗箱】：真正的核心链表藏在这里！
};

// drivers/base/base.h - “行业协会”暗箱 (不对外暴露)
struct subsys_private {
    struct kset subsys;                 // 负责在 /sys/class/ 下生成目录
    struct klist klist_devices;         // 【花名册】：挂载所有属于该类的 device
};

// include/linux/device.h - “具体的商家实体”
struct device {
    struct kobject kobj;                // 生成 /sys/devices/... 下的目录
    dev_t devt;                         // 【关键】：传给 udevd 去 mknod 的设备号
    struct class *class;                // 指向它所属的行业协会
    struct klist_node knode_class;      // 链表节点：用来把自己挂进 klist_devices
};
```

### 4. VFS 会话层 (Virtual File System)

C

```c
// include/linux/fs.h
struct inode {
    umode_t i_mode;                     // 文件类型
    dev_t i_rdev;                       // 【密码】：物理文件自带的设备号
    struct cdev *i_cdev;                // 缓存：open后指向查到的 cdev
};

struct file {
    const struct file_operations *f_op; // 【多态核心】：当前会话的操作指针
    void *private_data;                 // 【私有口袋】：给驱动存放上下文数据
};
```

# 🚀 动态演进：API 深度调用与物理突变流程

现在，我们以驱动的 `__init` 函数为起点，直到用户态触发 `open`，严格按照时间线拆解 API。

## 阶段一：向系统摇号占坑 (Namespace Allocation)

### API 1: `alloc_chrdev_region`

- **功能**：动态分配设备号，防止与其他驱动冲突。
    
- **原型**：`int alloc_chrdev_region(dev_t *dev, unsigned baseminor, unsigned count, const char *name);`
    
- **参数说明**：
    
    - `[OUT] dev`：内核返回的“营业执照号”（主设备号+起始次设备号）。
        
    - `[IN] baseminor`：请求的起始次设备号（通常为 0）。
        
    - `[IN] count`：需要连续占用几个次设备号（如 1）。
        
    - `[IN] name`：驱动名称（如 `"ipc-shm"`）。
        
- **底层第一性原理**：内核 `kmalloc` 一个 `char_device_struct`，填入 `name`，并将其塞进 `chrdevs[]` 数组。**此时没有任何路由能力。**
    

## 阶段二：装配 CDEV 并激活内部路由 (Internal Routing)

### API 2: `cdev_init`

- **功能**：将操作函数手册挂载到设备对象上。
    
- **原型**：`void cdev_init(struct cdev *cdev, const struct file_operations *fops);`
    
- **参数说明**：
    
    - `[IN/OUT] cdev`：你自己结构体里包含的 `cdev` 内存块。
        
    - `[IN] fops`：你写的 `open/read/write` 函数集。
        
- **底层第一性原理**：清理内存，并执行 `cdev->ops = fops;`。
    

### API 3: `cdev_add`

- **功能**：将设备号与设备对象绑定，注入内核查号系统。
    
- **原型**：`int cdev_add(struct cdev *p, dev_t dev, unsigned count);`
    
- **参数说明**：
    
    - `[IN] p`：初始化后的 `cdev`。
        
    - `[IN] dev`：阶段一摇号得到的设备号。
        
- **底层第一性原理**：内核生成一个 `struct probe` 节点。设置 `probe->dev = dev`，并**强行把 `probe->data` 指针指向你的 `cdev` 首地址**。最后挂入 `cdev_map` 哈希表。
    
- **费曼隐喻**：路由线已接通，但由于外面还没有设备节点，此时是“锦衣夜行”。
    

## 阶段三：LDM 外部挂牌与触发 Udev (Sysfs & /dev Creation)

### API 4: `class_create`

- **功能**：在内核 LDM 中注册一个“行业协会”。
    
- **原型**：`struct class *class_create(struct module *owner, const char *name);`
    
- **参数说明**：
    
    - `[IN] owner`：通常填 `THIS_MODULE`。
        
    - `[IN] name`：类名，如 `"ipc_shm_class"`。
        
- **底层第一性原理**：内核不仅 `kmalloc` 了 `struct class`，还秘密创建了 `struct subsys_private`，并**初始化了一条空的花名册链表 `klist_devices`**。同时在 `/sys/class/ipc_shm_class/` 生成目录。
    

### API 5: `device_create`

- **功能**：创建具体的设备实体，加入花名册，并广播让外界建文件！
    
- **原型**：`struct device *device_create(struct class *class, struct device *parent, dev_t devt, void *drvdata, const char *fmt, ...);`
    
- **参数说明**：
    
    - `[IN] class`：上一步创建的 `class` 指针。
        
    - `[IN] parent`：父设备（通常传 `NULL`）。
        
    - `[IN] devt`：**核心！把阶段一的设备号传进来！**
        
    - `[IN] drvdata`：私有数据（可传 `NULL`）。
        
    - `[IN] fmt`：设备名，如 `"ipc-shm"`。
        
- **底层第一性原理**：
    
    1. 生成 `struct device`。
        
    2. 将它的 `knode_class` 节点**链入 `class->p->klist_devices` 这个花名册链表中**。
        
    3. 设置 `dev->devt = devt`。
        
    4. **大爆发：调用 `kobject_uevent()` 发出 Netlink 广播。用户态的 `udevd` 收到消息后，提取出设备号和设备名，在屏幕外默默帮你执行了 `mknod /dev/ipc-shm c 244 0`。**
        
- **结果**：至此，`/dev/ipc-shm` 物理文件出现，且它的 `inode` 里写入了设备号！
    

## 阶段四：VFS 延迟绑定与驱动级 Open (The Grand Finale)

当用户应用执行 `open("/dev/ipc-shm", O_RDWR);`，VFS 的兜底函数 `chrdev_open` 拿着设备号去 `cdev_map` 查出 `cdev` 指针，并将 `file->f_op` 狸猫换太子，最后执行到了你写的代码：

### API 6: `my_dev_open` (驱动工程师的最终战场)

- **功能**：完成多实例上下文的绝对绑定。
    
- **原型**：`static int my_dev_open(struct inode *inode, struct file *file);`
    
- **输入参数说明**：
    
    - `[IN] inode`：VFS 传来的磁盘节点缓存，内部 `inode->i_cdev` 已经指向了你阶段二注册的基类 `cdev`。
        
    - `[IN/OUT] file`：VFS 为本次操作新建的动态会话对象“购物篮”。
        
- **底层第一性原理与代码实现**：
    

C

```c
// 假设你的大结构体长这样：
struct ipc_shm_device {
    void *hardware_base;
    int irq_num;
    struct cdev cdev;  // 基类嵌在里面
};

static int my_dev_open(struct inode *inode, struct file *file)
{
    struct ipc_shm_device *my_dev;

    // 第一性原理突变 1：指针运算魔法。
    // VFS 只知道基类 cdev 的地址，我们通过 container_of (利用编译期 offsetof)
    // 强行减去偏移量，反推算出整个大对象 struct ipc_shm_device 的首地址！
    my_dev = container_of(inode->i_cdev, struct ipc_shm_device, cdev);

    // 第一性原理突变 2：私有上下文装载。
    // 把算出来的大对象指针，塞进当前会话的口袋里。
    // 这意味着后续的 read/write 调用时，只要拿出 file->private_data，
    // 就能直接访问 my_dev->hardware_base，实现了完美的面向对象与多实例隔离！
    file->private_data = my_dev;

    return 0;
}
```

### 总结

这就是一条从 **占坑(chrdevs)** -> **通网线(cdev_map)** -> **挂招牌(LDM class/device)** -> **顾客上门接管上下文(VFS open)** 的铁血流水线。所有的复杂性，都是为了实现一个终极目标：**极致的解耦与多态**。
