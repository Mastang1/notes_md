_**总结：
1.LD将不同的类型的data放在不同的ram分区，避免masters之间的访问竞争；
2.TCM中放中断代码，避免DMA总线风暴波及中断**_

## Cortex-M 总线架构与仲裁机制核心笔记

### 一、 核心原理解析 (Core Principles)

- **多层矩阵架构 (Multi-layer Bus Matrix)**
    
    - 现代总线不是“单行道”，而是**网状交叉开关 (Crossbar Switch)**。
        
    - 多个主设备 (Master，如 CPU、DMA) 访问不同的从设备 (Slave，如 Flash、SRAM) 时，物理链路互不交叉，可实现 **100% 并行，零延迟**。
        
- **流水线交叠 (Pipelined Overlap)**
    
    - AHB 总线将一次传输分为**地址阶段**和**数据阶段**。
        
    - 当前设备在执行数据读写时，总线可以同步接收下一个设备的地址请求，极大提升吞吐量。
        
- **仲裁触发的绝对条件**
    
    - **只有当两个 Master 在同一时钟周期试图访问“同一个 Slave 的同一个物理接口”时，才会触发仲裁。**
        
    - 仲裁失败的一方，其 `HREADY` 信号会被总线矩阵拉低，导致**流水线被迫挂起 (Stall) 死等**，产生隐性的性能降级（即 CPU 表现为空闲，但执行极慢）。
        

### 二、 性能测试与调优避坑指南 (Pitfall Guide)

#### 💣 坑点 1：把鸡蛋放在同一个篮子里 (单块 SRAM 拥堵)

- **现象**：DMA 全速搬运外设数据到 SRAM 时，CPU 执行代码异常卡顿。
    
- **避坑指南 - SRAM 分块 (Banking)**：
    
    - 不要把变量和缓存全扔进默认的 RAM 区！高端 MCU 通常将内部 RAM 切分为多个独立的物理模块（如 `SRAM1`, `SRAM2`, `SRAM3`）。
        
    - **策略**：通过修改链接脚本 (`.ld` 文件 / 分散加载文件)，强制将 **DMA 的数据缓冲区**放在 `SRAM1`，将 **CPU 的全局变量/堆栈**放在 `SRAM2`。物理隔离后，仲裁彻底消失。
        

#### 💣 坑点 2：核心代码被 DMA 总线风暴波及

- **现象**：大量并发 DMA 操作导致总线极度拥堵，关键中断（如电机控制 PWM 中断）响应延迟大幅增加，甚至错过死区时间。
    
- **避坑指南 - 榨干 TCM (Tightly Coupled Memory)**：
    
    - TCM (ITCM 指令 / DTCM 数据) 是专属于 CPU 核的“私有领地”，**完全绕开系统级总线矩阵**。
        
    - **策略**：必须将**中断向量表 (Vector Table)**、**硬实时控制算法代码**、以及**核心任务栈 (Stack)** 强制分配到 TCM 中。这样无论外部总线如何发生风暴，核心控制逻辑永远不会被挂起。
        

#### 💣 坑点 3：忽略总线主设备的默认优先级

- **现象**：认为只要开了 DMA 就能提升性能，结果开启 DMA 后系统反而变慢。
    
- **避坑指南 - 检查 QoS 与仲裁权重**：
    
    - 硬件默认仲裁规则通常是 **DMA 优先级 > CPU 数据总线优先级**。
        
    - **策略**：如果 DMA 搬运的是不紧急的数据（如后台日志落盘），必须在 SoC 寄存器中调低该 DMA 通道的 QoS 权重，或者配置为“轮询 (Round-Robin) 仲裁”，防止低价值数据“饿死”高价值的 CPU 运算。
