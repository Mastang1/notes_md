# 🧠 【架构师笔记】Linux 设备树驱动开发与启动全景指南

## 模块一：核心认知与运行机制 (The "Why" & "How")

- **痛点起源**：为了终结将“硬件物理基地址”硬编码在 C 语言源码中的混乱（所谓 Board-specific trash），实现一套通用代码适配无数开发板。
- **本质**：一份**纯文本的“硬件资产交接单”**，将硬件物理拓扑与驱动代码彻底进行物理隔离。它定义了内核设备模型 (Linux Kernel Device Model) 的数据基础。

## 🌟 模块二：启动魔法 —— 设备树的“时空大流转”（新增）

在 C 语言代码运行前，设备树文本（`.dts`）是如何变成内存里的 `struct platform_device` 的？这是一个跨越三个运行时的接力赛：

- **第一棒：Bootloader (如 U-Boot) 的物理装载**
    - **动作**：U-Boot 将编译好的二进制设备树（`.dtb`）从 Flash 读入到 RAM 的某块固定物理地址中。
    - **交接**：U-Boot 在准备跳转启动 Linux 内核时，会将这个 `.dtb` 的物理首地址，强行塞进 **CPU 的特定寄存器** 中（例如 ARM32 架构下规定存放在 `r2` 寄存器，ARM64 在 `x0` 寄存器）。这是引导程序与内核的终极硬核契约。
- **第二棒：Linux 内核早期的“解压缩” (Unflattening)**
    - **动作**：Linux 内核刚启动（执行 `setup_arch`）时，会去读取那个特定的 CPU 寄存器，找到 `.dtb` 的地址。
    - **校验与展开**：内核先核对头部魔数（Magic Number `0xD00DFEED`），然后将紧凑的二进制数据，在内存中动态分配、展开为一棵由 **`struct device_node`** 组成的逻辑树（这叫 Unflatten Device Tree）。此时，硬件资产有了初步的内存形态。
- **第三棒：总线管理器的“捏造与挂牌” (Populate)**
    - **动作**：内核在进入驱动初始化阶段时，Platform Bus（平台总线）会遍历这棵 `device_node` 树。
    - **实例化**：只要节点里有 `compatible` 属性，且挂在根目录或 `simple-bus` 下，内核就会**自动 `malloc` 内存，生成大名鼎鼎的 `struct platform_device`（男嘉宾）**。
    - **融合**：内核将原本的 `device_node` 指针塞进 `platform_device.dev.of_node` 中，并将这个男嘉宾挂载到总线的匹配链表上。至此，文本文件彻底化身为可与 Driver 匹配的 C 语言对象！

## 模块三：数据元素与契约精神 (Bindings)

- **契约中心**：编写设备树绝不能凭感觉捏造属性名，必须严格遵守 Linux 内核源码 `Documentation/devicetree/bindings/` 目录下的 YAML 规范。
- **通用标准属性（必备）**：
    - `compatible`：寻址暗号（如 `"st,stm32-usart"`）。总线通过它进行 Match。
    - `reg`：物理基地址与长度（供驱动进行寄存器映射）。
    - `interrupts`：硬件逻辑中断号。
    - `status`：生死簿（`"okay"` 为启用，`"disabled"` 为物理切断不生成 device）。
- **契约审判**：如果漏写 Required（必填项），编译 `.dtb` 会报错；运行期驱动提取也会失败导致宕机。

## 模块四：驱动工程师的取件 API (Resource Extraction)

当总线匹配成功触发 `probe(struct platform_device *pdev)` 时，驱动工程师需使用 **Devres（自动垃圾回收，带有 `devm_` 前缀）** 神仙 API 提取资源：

1. **拿地契（内存映射）**：
    - `platform_get_resource(pdev, IORESOURCE_MEM, 0)` -> 获取物理地址信息。
    - `devm_ioremap_resource(dev, res)` -> 物理地址转虚拟地址指针（直接可读写）。
2. **牵网线（中断申请）**：
    - `platform_get_irq(pdev, 0)` -> 获取内核映射好的虚拟中断号。
    - `devm_request_irq(...)` -> 绑定中断服务函数 (ISR)。
3. **读私信（非标属性）**：
    - `device_property_read_u32(dev, "current-speed", &val)` -> 提取自定义业务参数（如波特率）。

## 模块五：面向对象的工程规范 (.dtsi vs .dts)

- **`.dtsi` (基类/SoC级)**：由芯片原厂维护。描述硅片内部不可变的物理外设拓扑。为了防止引脚冲突，所有外设必须默认 `status = "disabled";`。
- **`.dts` (子类/Board级)**：由下游驱动工程师维护。描述具体的绿色 PCB 板，通过 `#include` 继承基类。
- **修改法则（标签覆盖）**：绝对不要按原层级重新嵌套！使用 `&uart1 { status = "okay"; };` 的 Label 引用语法直接在底层进行跨层级追加或覆写。

## 🚀 模块六：架构深水区 —— 4 个致命的高阶知识点（深度展开版）

在真实的复杂工程中，仅仅会用 `reg` 和 `interrupts` 远远不够，以下 4 个知识点决定了你是否具备顶级驱动架构能力：

### 1. Pinctrl（引脚复用子系统）：解决“一个引脚，身兼数职”的物理冲突

- **底层痛点**：现代 SoC 的物理引脚非常匮乏，一个 PA9 引脚，既可以配成普通 GPIO，也可以配成 I2C_SCL，还能是 UART_TX。在驱动接管寄存器之前，谁来负责把硬件内部的 MUX（多路选择器）拨到正确的线路上？
- **工作机制**：设备树通过引用 Pinctrl 节点来解决。
    
    ```
    &uart1 {
        pinctrl-names = "default", "sleep";
        pinctrl-0 = <&uart1_pins_a>; // 正常工作时的引脚状态
        pinctrl-1 = <&uart1_sleep_pins>; // 休眠时的引脚状态
    };
    ```
    
- **深层意义**：在内核调用驱动的 `probe()` **之前**，Platform 总线会自动解析这些属性，调用 Pinctrl 子系统自动把物理引脚切换到 `uart1_pins_a` 定义的复用模式。驱动代码中根本不需要写任何管脚配置代码！

### 2. Phandle（句柄指针）：解决“跨越时空的硬件互相调用”

- **底层痛点**：设备树是一个层级树，但在物理主板上是网状连接。比如，我的 UART 控制器需要挂载某个时钟树 (RCC) 节点上获取频率，它怎么引用别的节点？
- **工作机制**：使用 `&` 引用。
    
    ```
    clocks = <&rcc 12>; // &rcc 是目标节点，12 是传递给该节点的参数
    ```
    
- **深层意义（编译器的魔法）**：在执行 `make dtbs` 编译时，编译器 (DTC) 会为 `rcc` 节点自动生成一个全局唯一的 32 位整型 ID（这叫 Phandle）。当内核解析到 `<&rcc>` 时，实际上是在内存树里用这个 ID 进行查找，完美实现了不同硬件节点之间的“指针引用”。

### 3. 特殊幽灵节点（`chosen` 与 `aliases`）：没有物理实体的桥梁

它们必须放在根节点 `/` 下，且没有任何 `compatible` 属性，不产生任何实体 Device。

- **`chosen` 节点（动态传参通道）**：
    - 它是 Bootloader 与 Linux 内核交互的“秘密情报站”。设备树是静态编译好的，但有些参数每次开机都变（如 MAC 地址），或者是由 U-Boot 决定的命令行参数（如 `bootargs = "console=ttyS0,115200 root=/dev/mmcblk0p2";`）。U-Boot 会在启动前，**动态修改内存中的 `.dtb`**，往 `chosen` 节点里注入这些字符串，内核启动后去这里读取。
- **`aliases` 节点（用户空间的锚点）**：
    - 如果板子上有 `serial@40011000` 和 `serial@50000000`，哪个是 `ttyS0`，哪个是 `ttyS1`？不能靠加载顺序，必须固化！
    - 在 `aliases` 里写上 `serial0 = &uart1;`，内核虚拟文件系统 (VFS) 解析后，就会雷打不动地把 `uart1` 绑定为 `/dev/ttyS0`，确保应用层代码永远不会串线。

### 4. 设备树插件 (DTO, Device Tree Overlays)：热插拔时代的终极解法

- **底层痛点**：传统的设备树在开机时就被 U-Boot 喂给内核，内核将其定型为内存树。那如果你用的是树莓派，开机状态下用户突然插上了一个 SPI 屏幕模块 (HAT) 怎么办？总不能重启系统并换个设备树吧？
- **工作机制**：内核引入了 Overlays 机制。开发屏幕模块的厂商，只需提供一个微型的碎片化设备树文件 `.dtbo`（编译后的 Plugin）。
- **深层意义**：在 Linux 运行期间，用户层可以通过 ConfigFS 接口将 `.dtbo` 加载进内核。内核的解析器会将其作为一根“小树枝”，**动态嫁接到当前正在运行的内存主树上**。主树一旦探测到新节点，总线立刻激活，实时触发新驱动的 `probe()` 绑定。这是极其优雅的硬件热加载解决方案！
