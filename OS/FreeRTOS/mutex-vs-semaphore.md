### 一、 核心概念：优先级反转 (Priority Inversion)

**优先级反转**是指在多任务抢占式操作系统中，系统调度的优先级顺序被破坏，导致**中等优先级任务先于高优先级任务执行**的一种异常状态。

**具体发生场景：**

1. **低**优先级任务 (Task L) 获取了某个共享资源（如信号量）。
    
2. **高**优先级任务 (Task H) 想要获取该资源，但因资源被占而进入阻塞态（挂入 Wait List）。
    
3. 此时，一个**中**等优先级的任务 (Task M) 就绪。由于 Task M 优先级高于 Task L，它抢占了 Task L 的 CPU 执行权。
    
4. **最终结果：** Task H 必须等待 Task L 释放资源，而 Task L 此时又被 Task M 剥夺了执行时间。实际上变成了 Task H（高）在等待 Task M（中）执行完毕，这严重破坏了 RTOS 的实时性和响应确定性。

### 二、 核心区别：互斥量 (Mutex) vs 普通信号量 (Semaphore)

从内核 TCB (任务控制块) 挂载链表的操作维度来看，两者的底层行为差异巨大。详细对比如下：

|**对比维度**|**普通信号量 (Semaphore)**|**互斥量 (Mutex)**|
|---|---|---|
|**内部数据结构**|仅包含“计数值”和“等待链表 (Wait List)”。|包含“等待链表”且强制绑定 **Owner 属性** (指向当前持有该资源的 TCB 指针)。|
|**所有权概念**|**无所有权**。任何任务或中断都可以 Give/Post 信号量。|**严格的所有权**。只有成功 Take 该 Mutex 的任务 (Owner) 才有权 Release 它。|
|**获取阻塞时**<br><br>  <br><br>_(TCB 链表行为)_|**常规切换**：仅将请求者的 TCB 从就绪链表 (`ReadyList`) 移入该信号量的 `WaitList`。持有者 TCB 不受任何影响。|**跨链表迁移 (优先级继承)**：请求者移入 `WaitList`。若发生优先级反转，内核会通过 Owner 找到持有者 TCB，将其从低级 `ReadyList` 强行拔出，**插入到与请求者同等级的高级 `ReadyList` 中**。|
|**释放资源时**<br><br>  <br><br>_(TCB 链表行为)_|**单向唤醒**：直接从 `WaitList` 中将优先级最高的 TCB 唤醒并移回 `ReadyList`。|**双向校验与恢复 (优先级恢复)**：<br><br>  <br><br>1. 校验当前请求释放的 TCB 是否为 Owner。<br><br>  <br><br>2. 若此前发生了继承，先将 Owner TCB 从高级 `ReadyList` 拔出，降级并**挂载回原有的低级 `ReadyList`**。<br><br>  <br><br>3. 唤醒 `WaitList` 中的等待者。|
|**任务删除时的**<br><br>  <br><br>**安全性机制**|任务被强杀时，TCB 销毁。其之前获取的信号量状态不变，等待该信号量的任务**可能永久死锁**。|健壮的 RTOS 在 TCB 中维护了一个“持有的 Mutex 链表”。任务被杀时，内核会**自动遍历并释放**这些 Mutex，唤醒等待者，保障系统状态机完整。|
|**核心使用场景**|侧重于**任务间的同步** (如：中断通知任务处理数据) 或 **资源计数**。|严格限制用于**临界区保护** (如：独占访问全局变量、硬件外设等)。|

---

**💡 笔记总结**

信号量的调度是**“单线”**的，只负责管理排队等待者的状态转换；而互斥量是**“双线联动”**的，它深度介入内核的调度策略，不仅管理等待者，还会为了维护系统的实时性（打破优先级反转）去主动修改持有者的系统优先级链表状态。
