### 一、 测试流程：单向发送还是 Ping-Pong 通信？

**结论是：吞吐量测试必须使用“单向流水线发送（One-way Streaming）”，绝对不能使用 Ping-Pong（乒乓）通信。**

- **Ping-Pong（RTT）是用来测延迟（Latency）的**：在 Ping-Pong 模型中，发送方发完一个包后，必须阻塞等待接收方回传 ACK 甚至完整 Payload。这意味着通信链路在绝大部分时间处于空闲等待状态，流水线被严重破坏，测出的带宽数据会极度失真。
    
- **单向流水线（One-way）是用来测吞吐量（Throughput）的**：发送方（Tx）像消防水龙带一样，只要共享内存的队列（Ring Buffer）有空位，就疯狂压入数据；接收方（Rx）只要看到队列有数据，就疯狂提取并释放空位。双方通过批量唤醒（Batched Notification）或者自旋锁解耦，最大化占满内存总线带宽。
    

### 二、 Payload 拷贝机制：收发双方需要 Copy 吗？

这是一个极其关键的分水岭。在业界标准的 IPC 吞吐量基准测试中，**拷贝（Deep-copy）和零拷贝（Zero-copy）是两个完全不同的测试用例，它们评估的系统瓶颈截然不同**。

一个严谨的测试矩阵必须包含以下两种场景：

#### 1. 零拷贝模式 (Zero-copy / Pointer Passing)

- **操作逻辑**：Tx 仅将数据在共享内存中的“地址指针”和“长度”放入队列；Rx 取出指针后，直接在共享内存上进行读写，处理完毕后释放该指针。**全程没有 `memcpy`**。
    
- **测试目的**：测试 IPC 框架本身的“管理开销”——即队列同步、锁竞争、通知机制（如跨核中断或 Event）的效率。
    
- **适用场景评估**：适用于流媒体、大文件映射等原生支持就地处理的业务。
    

#### 2. 深拷贝模式 (Deep-copy / App-level Copy)

- **操作逻辑**：
    
    1. Tx 从自己的 Local RAM (私有内存) 将 Payload 拷贝到 Shared Memory。
        
    2. 触发通知。
        
    3. Rx 从 Shared Memory 将 Payload 拷贝到自己的 Local RAM。
        
    4. 释放 Shared Memory。
        
- **测试目的**：测试系统最真实的端到端应用层带宽。因为在绝大多数复杂的业务软件中（如微服务、安全隔离域），应用层是不允许或者无法长时间锁定共享内存的，必须将数据拷贝到私有空间才能进行反序列化或复杂计算。
    
- **适用场景评估**：适用于 RPC 调用、Protobuf 数据序列化等标准控制流业务。
    

### 三、 对收发双方的严格要求

为了确保测出来的吞吐量是共享内存的真正极限，而不是被外部因素拖了后腿，收发双方必须满足以下条件：

1. **屏蔽外部 I/O 干扰**：测试期间，Tx 的数据源不能是从硬盘读取或网络接收的，必须是预先在内存中生成好的 Dummy Data（无意义填充数据）。
    
2. **解绑业务逻辑**：Rx 在收到数据后，只能做极轻量级的校验（如只检查头部 4 个字节的 Sequence ID），绝不能在测试中加入 CRC 校验、加密解密或复杂解析，否则你测出来的就成了 CPU 算力瓶颈，而不是 IPC 吞吐量。
    
3. **线程绑定与隔离 (CPU Affinity)**：收发线程必须被硬性绑定在不同的独立 CPU 核心上，防止操作系统调度器将它们放在同一个核心上进行时间片轮转，导致缓存频繁失效（Cache Thrashing）。
    
4. **背压处理能力 (Backpressure)**：Tx 不能因为队列满了就崩溃，必须具备优雅的自旋等待（Spin-wait）能力；Rx 必须具备批量消费（Batching）能力，即一次中断唤醒尽可能多地处理积压的数据包。
    

### 四、 吞吐量指标是如何定义的？

吞吐量的评估通常由两个核心维度的数学指标构成：

#### 1. 消息吞吐率 (Message Rate)

主要用于评估小包（如 8~64 Bytes）传输时的系统并发处理极限。衡量单位是 **Msg/s (Messages per Second)** 或 **IOPS**。

$$Message Rate = \frac{N_{total\_messages}}{T_{end} - T_{start}}$$

#### 2. 数据吞吐带宽 (Data Bandwidth)

主要用于评估大包（如 1MB~10MB）传输时的内存总线极限。衡量单位是 **MB/s** 或 **GB/s**。

计算公式为：

$$Bandwidth = \frac{N_{total\_messages} \times S_{payload\_size}}{T_{end} - T_{start}}$$

- _注意：严格的带宽计算通常只统计 Payload Size（有效载荷），不包含队列头部（Header）、指针等管理数据的开销，这被称为**有效吞吐量 (Goodput)**。_
