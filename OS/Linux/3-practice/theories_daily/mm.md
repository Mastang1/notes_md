> [!abstract] 核心认知基石：Linux 内核的两大“幻觉”制造机 在 MCU/RTOS 的平坦世界（Flat Memory）里，所见即所得，任务按优先级严格排队。而 Linux 内核本质上是一个“大忽悠”，它通过**虚拟内存**给每个进程制造了“独占海量连续内存”的空间幻觉，又通过**进程调度**制造了“独占 CPU”的时间幻觉。

---

### 🗺️ 第一部分：虚拟内存管理（拆穿空间幻觉）

**1. MMU 与多级页表：内核的“邮政重定向局”**

- **认知锚点**：抛弃 MCU 的物理直址思维。
- **费曼隐喻**：应用程序拿到的所有指针都是“假门牌号”。每次 CPU 读写时，硬件 MMU 都会悄悄查阅内核维护的“密码本（多级页表）”，把假门牌号重定向到真实的物理 DDR 仓库。
- **Lazy Allocation（空头支票）**：用户态 `malloc` 1GB 内存，内核其实一字节都没给，只给了一张“空头支票”。直到代码第一次写入引发**缺页异常 (Page Fault)**，内核才火速在物理内存中找空地填上，修改页表，然后让 CPU 继续。

**2. 内核态内存分配双雄：`kmalloc` vs `vmalloc`**

- **`kmalloc` (物理连续)**：分配“物理地址绝对挨在一起”的连排仓库。数量有限，**专供 DMA 使用**（因为 DMA 这种底层硬件不懂虚拟地址，只认物理直址）。
- **`vmalloc` (虚拟连续)**：分配“虚拟地址连续，但物理地址散落全国各地”的拼凑包厢。适合分配巨大的内存块，但**绝不能用于 DMA 搬运**。

### 🚦 第二部分：进程调度与生死时速（拆穿时间幻觉）

**1. 进程控制块：`task_struct` (终极户口本)**

- 相比于 RTOS 轻量级的 TCB（任务控制块），Linux 的 `task_struct` 极其庞大。它不仅存着栈指针，还提着进程的“全部家当”：专属的页表密码本、打开的文件描述符（VFS 会话）、信号处理机制等。

**2. CFS 完全公平调度器 (共产主义厂长)**

- **费曼隐喻**：不同于 RTOS 的“高优先级绝不让出 CPU”，CFS 调度器是一个极端追求公平的厂长。它用一棵**红黑树**记录每个进程干活的“虚拟运行时间 (`vruntime`)”。谁干的时间最少，下次就挑谁上 CPU，确保宏观上的时间平摊。

**3. 生死红线：`GFP_KERNEL` vs `GFP_ATOMIC`** 这是驱动开发中引发 Kernel Panic（内核崩溃）的头号杀手！

- 🟢 **`GFP_KERNEL` (可睡眠，进程上下文)**：如果内存不够，当前进程挂起睡觉，**主动交出 CPU 控制权给 CFS 调度器**。等别人释放内存了再被唤醒。适用于驱动的 `open/read/write` 阶段。
- 🔴 **`GFP_ATOMIC` (急救模式，中断上下文)**：如果内存不够，**绝对不睡觉，立刻返回 NULL**。专供硬中断 (Hard IRQ)、软中断 (Soft IRQ) 等**没有 `task_struct`、不能被 CFS 调度**的上下文使用。如果在中断里睡眠，内核将永远不知道如何唤醒它，直接死机！

### 🎯 第三部分：架构师级自测灵魂三问 (复盘验收)

在笔记末尾，请用这三个问题冷酷自测（大厂高频面试题）：

1. **链路回放**：用户态 `malloc` 申请 1GB 内存后第一次写入时，硬件 MMU 和内核到底发生了什么？（必须答出：空头支票 -> MMU 报错 -> Page Fault 陷入 -> 申请物理页 -> 修改页表）。
2. **选型决策**：在驱动里申请 2MB 内存给网卡芯片做 DMA 搬运，应该用 `kmalloc` 还是 `vmalloc`？（必须答出：`kmalloc`，因为 DMA 是纯物理硬件，不懂页表，必须物理地址连续）。
3. **生死边界**：在 GPIO 按键的中断处理函数 (ISR) 中，动态申请小块内存传了 `GFP_KERNEL` 会发生什么？（必须答出：中断上下文无 `task_struct`，引发 scheduling while atomic 崩溃，必须用 `GFP_ATOMIC`）。
