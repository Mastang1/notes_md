_**个人理解：Q:我理解：memcpy_io的访问方法，也不能简单的叫做对齐访问，它是根据是否对其的内存，执行不同的底层指令；对吗?**_

> A:**完全正确！你总结得极其精准，简直是一语道破天机。**

`memcpy_toio` 并不是强制要求你传入的地址必须绝对对齐（如果你传入不对齐的地址，它并不会像 `memcpy` 那样直接引发系统崩溃），而是它内部具备“自适应降级探测”的智慧。

它就像一个极其严谨的搬运工，在动手之前和搬运过程中会不断判断：

* **遇到对齐的块**：如果当前地址刚好是 4 字节或 8 字节对齐，且剩余数据量够大，它就使用 4 字节或 8 字节的指令（如 `writel` / `writeq`）来稍微提高一下吞吐率。
* **遇到未对齐的地址或零头**：一旦发现起始地址偏了，或者最后只剩下两三个字节，它立刻“认怂降级”，切成最底层的单字节指令（`writeb` / `strb`）逐个搬运。

**核心逻辑在于：单字节访问在硬件总线上永远不存在“非对齐”的概念**（1 个字节本身就是最小操作粒度）。

所以，你说得非常对！它不是“只做对齐访问”，而是**根据内存地址的实际对齐情况，动态选择最安全、符合设备内存严苛物理属性的底层指令**。这就是它能保命的根本原因。


_**注意：可以在栈空间执行快速处理，然后通过memcpy_toio执行写入**_

## 1. 深度剖析：Linux 内核内存映射与物理属性全景图

在 ARM64 架构下，Linux 内核通过页表（PTE）和 `MAIR_EL1`（内存属性间接寄存器）来精细控制每一块虚拟内存的物理属性。我们可以将它们从“最严苛/最慢”**到**“最自由/最快”排列。

理解的核心密码是三个字母：

- **G (Gathering)**: 是否允许合并总线写请求（比如把两次单字节写合并成一次双字节写）。
    
- **R (Reordering)**: 是否允许 CPU 乱序执行读写指令。
    
- **E (Early Write Ack)**: 是否允许在数据还没真正到达外设时，中间的 Write Buffer 就提前回复 CPU“写完了”。

## 2. API 备忘录：Device Memory vs Normal Memory 访问指南

既然内存属性分了阶级，Linux 内核自然准备了两套截然不同的工具箱。**永远不要混用！**

### 🛡️ 武器库 A：操作 IO/设备内存 (Device Memory)

**核心法则**：必须使用专属 API，保证时序，防止编译器和 CPU 优化，安全处理对齐。

- **单字/标量读写（最常用）：**
    
    - `readb(addr)` / `writeb(val, addr)`: 8-bit (单字节) 安全读写。
        
    - `readw(addr)` / `writew(val, addr)`: 16-bit 安全读写。
        
    - `readl(addr)` / `writel(val, addr)`: 32-bit 安全读写。
        
    - `readq(addr)` / `writeq(val, addr)`: 64-bit 安全读写。
        
    - _笔记_：这些函数底层不仅用了正确的汇编指令（如 `str`），还往往隐式包含了**内存屏障 (Memory Barrier)**，确保代码顺序就是总线电平的顺序。
        
- **块搬运与填充（防宕机必备）：**
    
    - `memcpy_toio(dest_io, src_normal, size)`: 往设备内存写块数据。带非对齐降级机制。
        
    - `memcpy_fromio(dest_normal, src_io, size)`: 从设备内存读块数据。
        
    - `memset_io(dest_io, val, size)`: 安全地格式化设备内存。
        

### 🏎️ 武器库 B：操作普通内存 (Normal Memory)

**核心法则**：追求极致性能，默认编译器和硬件会处理好一切优化和对齐。

- **标量读写：**
    
    - 直接使用 C 语言指针解引用：`*ptr = val;` 或 `val = *ptr;`
        
- **块搬运与填充：**
    
    - `memcpy(dest, src, size)`: 标准 C 库的内核汇编极致优化版。大量使用向量/成对寄存器。
        
    - `memset(dest, val, size)`: 同上，飞快。
        
    - `memcmp(ptr1, ptr2, size)`: 内存比较。
        

## 3. 核心思想沉淀：你对两种内存访问的深刻理解

（以下内容，非常适合写在你的博客或技术分享的结尾）

> 在 Linux 内核开发中，导致 Kernel Panic (Data Abort) 的最隐蔽杀手，往往源于对**“指针阶级（内存属性）”**的无知。
> 
> 1. **对 Device Memory（由 `ioremap` 映射）的访问，必须是“敬畏且保守”的。** > 这种内存直连物理外设，它没有 Cache 的缓冲，没有乱序的容忍度，更对地址的物理对齐有着类似强迫症般的严苛要求。因此，必须使用 `memcpy_toio` 等带有 `_io` 后缀的函数。它们是**“按规矩办事的保守派”**：在执行前会像雷达一样探测地址的对齐情况，一旦发现不对齐，绝不莽撞，而是主动降级为最基础的单字节（`strb`）指令，虽然牺牲了带宽，但保证了总线交互的绝对合法与系统生存。
>     
> 2. **对 Normal Memory（由 `kmalloc`、栈分配或 `memremap` 映射）的访问，则是“激进且奔放”的。**
>     
>     这种内存是纯粹的存储介质，有强大的 Cache 体系和自动非对齐处理的 MMU 硬件作为后盾。标准 C 库函数 `memcpy` 是一个**“蒙眼狂奔的飙车党”**，它的设计哲学是榨干每一滴 CPU 性能。它不会去探测地址是否对齐，而是直接使用 16 字节甚至更大的向量批量读写指令（如 `stp/ldp`）呼啸而过。
>     
> 
> **悲剧的发生，永远是因为让“飙车党”驶入了“崎岖严格的设备小路”。** 把指向 Device Memory 的未对齐指针强行塞给 `memcpy`，就如同让一辆 F1 赛车全速冲上未铺砌的碎石路，等待你的只有瞬间的车毁人亡（Kernel Panic）。
