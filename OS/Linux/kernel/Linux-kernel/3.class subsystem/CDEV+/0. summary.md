### 2. 划清界限：设备对象是如何生成文档节点并加入 `cdev_map` 的？

#### 步骤 B：在文件系统挂牌生成 `inode`（你笔记中的 Phase 3 阶段）

既然 `cdev_add` 不建文件，那 `/dev/ipc-shm` 这个文件（以及它背后的 `inode`）是怎么凭空冒出来的呢？

这里有两条路可以走：

1. **石器时代的土办法（手动）：**
    
    开发者在用户态的黑框框里，手动敲命令：`mknod /dev/ipc-shm c 244 0`。
    
    这句话的意思是：让 VFS 在磁盘上建一个文件，类型是字符设备(c)，强行在它的 `inode` 的 `i_rdev` 变量里写入 244:0 这个设备号。
    
2. **现代 Linux 的自动化魔法（LDM + udev）：**
    
    为了不让人手动敲命令，Linux 引入了设备模型 (LDM)。你在驱动代码里调用 `device_create(..., dev_id, ..., "ipc-shm")`。
    
    - **注意！** 这个函数也**不是**去建文件的！它只是向用户态广播了一条大喇叭（广播事件 `uevent`）：“系统里新来了一个设备，叫 ipc-shm，号码是 dev_id！”
        
    - 用户空间有一个后台进程叫 **`udevd` (或 mdev)**。它听到了广播，立刻接管工作，在后台**自动帮你调用了 `mknod`**，在 `/dev/` 下生成了那个文件！
