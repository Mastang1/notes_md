### 二、 源码级拆解：`device_create` 做了什么？

从内核源码第一性原理出发，`device_create()` 是一个极其厚重的包装函数（Wrapper），它的完整函数签名如下： `struct device *device_create(struct class *class, struct device *parent, dev_t devt, void *drvdata, const char *fmt, ...);`

它的执行流程严格分为以下四个物理突变阶段：

#### 1. 物理内存分配与基础填表 (Alloc & Init)

内核在堆内存中 `kzalloc` 出一个全新的 `struct device` 对象。随后把传入的 `devt`（你在 `alloc_chrdev_region` 摇号得到的设备号）写入 `dev->devt`。这就好比领了一张崭新的空白档案表，填上了你的营业执照号。

#### 2. 认祖归宗与行业归属 (Class & Parent Binding)

将 `dev->class` 指向你传入的 `struct class`（行业协会）。如果有父设备，就将 `dev->parent` 指向父设备，从而在内存中构建出庞大的设备拓扑树。

#### 3. 核心大杀器：`device_add()` (注册与 Sysfs 具象化)

这是最沉重的一步。它调用底层的 `kobject_add()`，在内核的虚机文件系统 `/sys/` 下动态生成代表该设备的物理目录（例如 `/sys/class/my_class/ipc-shm/`）。 同时，将这个 `struct device` 的链表节点 `knode_class`，死死地挂入行业协会的花名册 `class->p->klist_devices` 链表中。

#### 4. 跨界大喇叭：Netlink 广播 (`kobject_uevent`)

`device_add` 的尾声，会调用 `kobject_uevent(&dev->kobj, KOBJ_ADD)`。这会在内核态通过 Netlink 套接字向用户态发射一段包含 `MAJOR`（主设备号）、`MINOR`（次设备号）和 `DEVPATH` 的文本广播。

### 第四层：第一性原理 —— 为什么要这么折腾？

你可能会拍桌子问：“Linux 疯了吗？内核自己建一个 `inode` 只要几微秒，为什么要绕一圈让用户态的 `udevd` 去调用 `mknod`？”

这是基于强大的 **工程可控性** 考量：

1. **权限控制与安全隔离：** 如果内核自动生成 `/dev/ipc-shm`，它的权限默认是什么？`0777` 吗？那黑客随便一个普通用户就能读取你的底层共享内存了。内核不知道你的业务需求，只有用户态的配置文件（`udev rules`）知道这个设备该归属于哪个用户组、具备什么读写权限。
2. **动态命名机制（Persistent Naming）：** 假设你插了两个一模一样的 U 盘，内核只会无脑地叫它们 `sda` 和 `sdb`。下次重启插拔顺序换了，`sda` 就变成了 `sdb`，这会导致挂载灾难。而用户态的 `udevd` 可以在建 `inode` 前，读取 U 盘的 UUID 序列号，然后发起 `mknod` 给它们建一个永远不变的软链接（比如 `/dev/disk/by-uuid/1234-5678`）。
3. **微内核思想的延伸：** 尽量把“策略”移出高权限的内核态，保持内核的极简。

_(补充背景说明：在极新的 Linux 系统中，内核其实引入了一个叫 `devtmpfs` 的优化，它允许内核在发送 uevent 广播的**同时**，先在 `/dev` 下“垫”一个默认权限的 `inode`。但 `udevd` 收到广播后，依然会去调整这个 `inode` 的权限或重命名。所以“用户态接管策略”的核心架构思想和生命周期映射逻辑，至今未变！)_

**你的心智模型闭环：** `inode` 是内核的，没错；`/dev/` 路径也是内核文件系统管理的，没错。但它们的**“催生婆”**，是用户态的那个守护进程 `udevd` 拿着大喇叭里的信息，通过系统调用 `mknod` 强制要求内核生下来的！
