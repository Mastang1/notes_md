
### 【问题要点提取】

1. **意图分析**：你已经完全理解了设备树（DT）作为“物理硬件契约”的本质，并且明白内核会在编译期校验，并在运行期自动生成 `device` 变量（具体通常是 `platform_device`）。现在你的焦点转移到了**“落地开发”**：作为写驱动（Driver）的人，当总线把这个自动生成的 `device` 抛给我的 `probe` 函数时，我究竟该调用哪些核心 API 把资源“抠”出来？
2. **核心要点**：现代 Linux 驱动提取资源的核心场景分为三类：**内存映射（获取寄存器）、中断申请（获取 IRQ 线）、私有属性提取（读取自定义参数）**。
3. **架构师认知延伸**：内核黑客为了防止驱动工程师写出“内存泄漏”的代码，引入了 **Devres (Managed Device Resource)** 机制。现在的资源获取 API 通常都带有 `devm_` 前缀，它实现了资源的“自动垃圾回收”。

### 【展开讲解：资产交接大会 —— 驱动工程师的“取件指南”】

在 Linux Platform 驱动模型中，当总线通过 `compatible` 字符串匹配成功，调用你的 `probe(struct platform_device *pdev)` 函数时，相当于红娘把你领到了这块物理芯片面前。

`pdev` 这个指针就是设备树节点在内存中的化身。接下来，你需要使用内核提供的一套标准 API 来进行“资产提取”。

#### 一、 常用核心 API 盘点与场景描述

**场景 1：拿地契（提取并映射物理内存/寄存器）** 在 Linux 中，驱动绝对不能直接对物理地址（如 `0x40011000`）进行指针赋值读写，必须将其映射为虚拟地址（MMU）。

- **第一步：找物理地址**
    - **API**: `struct resource *platform_get_resource(struct platform_device *pdev, unsigned int type, unsigned int num)`
    - **描述**: 从 `pdev` 中提取设备树里 `reg` 属性定义的信息。`type` 填 `IORESOURCE_MEM`，`num` 填索引（比如 `0` 代表第一个 `reg` 地址段）。它返回的是纯物理的起止地址。
- **第二步：物理转虚拟 (IoRemap)**
    - **API**: `void __iomem *devm_ioremap_resource(struct device *dev, const struct resource *res)`
    - **描述**: 这是极其高频的神仙 API！它把刚才获取的物理 `resource` 映射为 CPU 可以直接读写的虚拟地址指针。**注意 `devm_` 前缀**，这是 Devres 机制（Managed Device Resource），意味着当设备被拔出或驱动卸载时，内核会自动释放这块虚拟内存映射，再也不用担心忘记写 `iounmap` 导致内存泄漏了！

**场景 2：牵网线（提取并注册硬件中断）**

- **第一步：找中断号**
    - **API**: `int platform_get_irq(struct platform_device *pdev, unsigned int num)`
    - **描述**: 从设备树的 `interrupts` 属性中提取内核能识别的逻辑中断号（Virtual IRQ number）。
- **第二步：绑定中断服务函数 (ISR)**
    - **API**: `int devm_request_irq(struct device *dev, unsigned int irq, irq_handler_t handler, unsigned long irqflags, const char *devname, void *dev_id)`
    - **描述**: 拿着刚才的 `irq` 号，向内核注册你的中断回调函数（`handler`）。同样带有 `devm_` 前缀，卸载时自动注销中断。

**场景 3：读私人信件（提取自定义/非标属性）** 有时我们需要获取设备树里的 `status` 之外的自定义参数，比如 `current-speed = <115200>;`。

- **API**: `int device_property_read_u32(struct device *dev, const char *propname, u32 *val)`
- **描述**: 直接通过属性名字符串（如 `"current-speed"`），去设备树节点里把整数值读到 `val` 变量中。

#### 二、 费曼式 Demo 拆解：一个 Serial 串口的完整 `probe` 流程

我们把设备树的“文本图纸”和 C 语言的“提取代码”放在一起对照，看看驱动工程师到底是怎么干活的。

**1. 硬件工程师写的图纸 (Device Tree)**

```
uart1: serial@40011000 {
    compatible = "st,stm32-usart";  // 暗号
    reg = <0x40011000 0x400>;       // 内存资源
    interrupts = <37>;              // 中断资源
    current-speed = <115200>;       // 业务自定义资源
};
```

**2. 驱动工程师写的接管代码 (C 语言 Probe 函数)**

```c
/* 自定义的串口结构体，用来保存提取出来的资源 */
struct my_uart_port {
    void __iomem *base_addr; // 映射后的虚拟基地址
    int irq;                 // 中断号
    u32 baudrate;            // 波特率
};

/* 核心交接现场：Match 成功后，总线强制调用此函数 */
static int my_uart_probe(struct platform_device *pdev)
{
    struct device *dev = &pdev->dev;
    struct my_uart_port *port;
    struct resource *res;
    int ret;

    // 0. 使用 devm_kzalloc 自动分配内存，存放私有数据
    port = devm_kzalloc(dev, sizeof(*port), GFP_KERNEL);
    if (!port) return -ENOMEM;

    // ==========================================
    // 场景 1：提取寄存器内存并映射
    // ==========================================
    // 从设备树 reg 属性中拿物理地址范围
    res = platform_get_resource(pdev, IORESOURCE_MEM, 0);
    // 将 0x40011000 映射为虚拟地址，交给 port->base_addr
    port->base_addr = devm_ioremap_resource(dev, res);
    if (IS_ERR(port->base_addr)) {
        dev_err(dev, "内存映射失败\n");
        return PTR_ERR(port->base_addr);
    }

    // ==========================================
    // 场景 2：提取中断号并注册回调
    // ==========================================
    // 从设备树 interrupts 属性中获取中断号 (37)
    port->irq = platform_get_irq(pdev, 0);
    if (port->irq < 0) return port->irq;

    // 申请中断，my_uart_isr 是你写的实际收发数据的硬件中断处理函数
    ret = devm_request_irq(dev, port->irq, my_uart_isr, 0, "stm32_uart", port);
    if (ret) {
        dev_err(dev, "申请中断失败\n");
        return ret;
    }

    // ==========================================
    // 场景 3：提取非标自定义属性
    // ==========================================
    // 尝试读取 current-speed 属性
    ret = device_property_read_u32(dev, "current-speed", &port->baudrate);
    if (ret) {
        // 如果设备树没写，给个默认值
        port->baudrate = 9600;
    }

    // ==========================================
    // 终极动作：向 Class 子系统 / VFS 注册
    // ==========================================
    // 资源齐备了，向系统暴露 operations，并把 port 私有数据绑定进去
    // (注册 struct file_operations 或 uart_ops，此处略过冗长注册代码)

    dev_info(dev, "串口初始化成功！波特率:%d, 映射地址:%p\n", port->baudrate, port->base_addr);

    // 把我们的私有结构体存到 pdev 中，方便后续 remove 等函数使用
    platform_set_drvdata(pdev, port);

    return 0; // 牵手彻底成功！
}
```

### 🧠 架构师心智模型最终缝合

仔细观察上面的 Demo，你会发现驱动开发的一个绝美之处：**代码里没有任何一个绝对的数字魔数（Magic Number）**。

`0x40011000`、`37`、`115200` 这些硬件物理强绑定的数字，全部被隔离在了设备树里。 你写的驱动代码完全是一套**“通用的数据提取与动作编排算法”**。哪怕明天这颗芯片改款了，串口地址变成了 `0x50000000`，你只要修改 `.dts` 文本文件并重新编译设备树，这份 `.c` 驱动代码**一行都不需要改**！
